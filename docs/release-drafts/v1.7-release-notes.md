# darklab_shell v1.7 Release Notes

v1.7 makes session files, command discovery, and terminal autocomplete feel much more complete. Files can be moved and renamed from the UI or terminal, workspace paths complete more like a real shell, and supported commands now have an app-native details view instead of living only as FAQ chips.

For the full list of changes included in this release, see the v1.7 section in [CHANGELOG.md](../../CHANGELOG.md).

## Highlights

### Files Can Move

Session files and folders can now be moved or renamed without leaving darklab_shell.

- Use the Files modal row action to move a file or folder.
- Drag a file onto a folder row to move it there.
- Run `file move <source> <destination>` from the terminal.
- Use `mv <source> <destination>` when you want the shorter shell-style form.

Moves stay inside your current session workspace. The app rejects unsafe paths, folder self-moves, symlinks, and accidental overwrites.

### File Commands Understand Simple Patterns

Common workspace commands now support predictable `*` patterns:

```bash
file ls darklab-*
mv darklab-* darklab/
file delete scan-*
```

The pattern is intentionally small: `*` matches inside one path segment and does not cross `/`. If a move matches more than one item, the destination must already be a folder. Deletes still ask for confirmation, and folders still require `-r` or `-rf`.

### Workspace Autocomplete Feels More Like a Shell

File autocomplete is now directory-aware. Commands such as `cat darklab/`, `cat ../`, and `cat ../darklab/` suggest direct children from that folder while preserving the path style you typed.

Folders now show and insert with a trailing `/`, and accepting a folder immediately refreshes autocomplete for the next path segment. `file move` and `mv` also understand source and destination slots, so the second path suggests useful destination folders instead of repeating the source.

### Command Details Are Easier to Find

Every supported external command now has a short description in `commands.yaml`. Users can run `commands info <command>` from the terminal or click a supported-command chip in the FAQ to open a details modal with the description, examples, flags, and subcommands pulled from the same registry.

### Smoother Active-Run UX

Active runs are less likely to surprise the user:

- Autocomplete no longer appears from the hidden prompt while the active tab is running a command.
- Choosing **Keep running** when closing a running tab keeps that run detached after browser reload instead of restoring it as a terminal tab again.
- Status Monitor uptime now counts upward locally between `/status` polls, so the dashboard feels live without making extra requests.

## Reliability Fixes

- Tool-created workspace folders, such as config folders from tools using session-scoped `XDG_CONFIG_HOME`, are repaired and shown in Files instead of causing read/list errors.
- Oversized workspace files now show the toast error before opening the viewer/editor modal, so the modal does not get stuck in a loading state.
- Placeholder autocomplete rows such as `<file>` and `<domain>` are now treated as guidance only. Tab, Enter, arrows, mouse, and touch skip those rows and operate on real suggestions.
- Duplicate `file list` and `file ls` autocomplete rows were removed.

## Testing

The release updates backend, Vitest, and Playwright coverage for workspace moves, glob list/move/delete flows, directory-aware autocomplete, command catalog data, oversized-file handling, active-run input blocking, detached-run reload behavior, and Status Monitor uptime ticking.

# darklab_shell v1.6 Release Notes

v1.6 is a big usability and reliability release. The headline is simple: darklab_shell now has real session files, safer scanner file handling, saveable workflows, stronger active-run recovery, and a much more useful Status Monitor.

If you only skim one part, start with these changes:

- **Session Files are now a first-class feature.** Users can create, view, edit, download, delete, browse, and autocomplete workspace files without giving scanner tools broad shell filesystem access.
- **Scanner commands can safely read and write session files.** Declared flags for tools like `nmap`, `curl`, `ffuf`, `nuclei`, `gobuster`, `katana`, `pd-httpx`, `subfinder`, `naabu`, `sslscan`, `testssl`, `nikto`, and `wpscan` are rewritten into guarded workspace paths.
- **Workflows are now user-saveable playbooks.** Built-in workflows remain, and users can create, edit, delete, inspect, and run their own session-scoped workflows from the Workflows panel or the `workflow` command.
- **Active runs survive browser changes much better.** Runs now stream through the `/runs` path with Redis-backed replay in production, same-session browsers can attach to live output, and closing a tab can detach instead of killing the backend process.
- **Status Monitor is now a real app surface.** It brings together active runs, CPU/memory telemetry, system/session cards, run activity visuals, attach/kill actions, and richer history insights.
- **Command metadata is much less fragmented.** Policy, autocomplete, examples, smoke coverage, runtime adaptations, file flags, wordlist slots, domain slots, and workflow coverage now share the `commands.yaml` registry model.

These notes focus on what matters most when using or operating v1.6. For the full list of changes included in this release, see the v1.6 section in `CHANGELOG.md`.

## Session Files and Safer Scanner Output

v1.6 introduces app-managed per-session workspaces. Files stay inside a validated session boundary with quota checks, safe relative paths, symlink protection, cleanup by inactivity, and operator-controlled enablement. Workspaces are still disabled by default until an operator turns them on.

The new Files experience includes:

- folder browsing with breadcrumbs
- new folder and new file flows
- edit, download, delete, and count-aware folder deletion
- manual refresh for command-generated files
- workspace-file autocomplete once the file list is loaded
- a dedicated file viewer modal with line numbers and search

The viewer handles common scanner output formats directly:

- JSON and XML formatting
- JSONL/NDJSON record-by-record formatting
- CSV/TSV preview tables
- raw HTTP response previews with status, headers, and body sections
- large text previews with loading feedback, bounded rendering, debounced search, lazy highlighting, and raw mode

Terminal file helpers are also available when Files are enabled:

- `file list`, `file show`, `file add`, `file edit`, `file download`, and confirmed `file rm`
- cwd-aware `cd`, `pwd`, `ls`, `ls -l`, `cat`, `mkdir`, and confirmed `rm`
- per-tab workspace current directories with `/` as the workspace root
- compact desktop and mobile prompt/path display

Scanner integrations now use declared file flags instead of broad path access. Workspace-aware flags are rewritten only when `commands.yaml` says they should be. That makes common command shapes useful while keeping the filesystem boundary tight.

ProjectDiscovery tools also get session-scoped runtime state and secondary output surfacing. Config, resume, response, screenshot, field, export, and log paths for tools such as `katana`, `pd-httpx`, `nuclei`, `subfinder`, and `naabu` can appear in Files under session-owned folders instead of disappearing into `/tmp` or tool-private config paths.

nmap now defaults toward TCP connect scan behavior where needed, avoiding the raw-socket failures that made `-sS` unreliable in the non-root scanner environment.

## Command Registry, Autocomplete, and Wordlists

v1.6 moves the visible external-command model into `app/conf/commands.yaml`. This reduces drift between what the app allows, suggests, documents, tests, and runs.

The registry now feeds:

- command policy and catalog metadata
- autocomplete examples and value types
- runtime adaptations such as injected flags, wrappers, managed config dirs, and environment changes
- smoke-test examples and seeded history
- workflow command coverage
- workspace-aware file flag declarations
- domain, wordlist, and restricted-input validation slots

Autocomplete is also smarter. It now ranks exact, prefix, token-boundary, substring, and fuzzy matches across command roots, subcommands, flags, arguments, workspace files, recent domains, and wordlists. File and wordlist completion can match useful path segments instead of only matching the beginning of a path.

Recent domains are now session-backed. When a flag or argument declares `value_type: domain`, the browser captures recent domains for the active session token and suggests them in future domain slots, including from another browser or private window using the same token.

SecLists support is catalog-backed too. The new `wordlist` command lists curated installed wordlist categories, can search entries, prints copy-friendly paths, and supports `wordlist --all` when you need the full scanned tree. Command metadata can declare `value_type: wordlist` plus `wordlist_category`, so autocomplete stays relevant instead of dumping every installed list everywhere.

Operators can now configure `restricted_command_input_cidrs` to block literal IP/CIDR inputs in declared target slots. The same validation path also checks app-managed workspace target-list files when declared read flags point at them. Domain names are intentionally not DNS-resolved in this first pass.

## Saveable Workflows

Workflows graduated from built-in guidance into user-saveable playbooks.

Users can now:

- create, edit, and delete session-scoped workflows
- define structured workflow inputs
- run workflows from the Workflows panel
- run workflows from the terminal with `workflow list`, `workflow show <name>`, and `workflow run <name> --input value`
- fill missing required inputs through transcript prompts

Built-in workflows still provide ready-made recon sequences, and two Files-backed examples show how workflows can generate session files and feed them into follow-up tools through declared workspace-aware flags.

## Active Runs, Reload Recovery, and Long Output

Run handling is much more resilient in v1.6.

Command starts now use the `/runs` streaming foundation. Production deployments can use Redis-backed replayable run events, while local development still has an in-process fallback. If the browser reloads or another same-session browser opens, the app can replay available output and continue streaming live events.

Active-run ownership is now browser-aware. Each browser gets a local client id, and `/history/active` can tell whether a run belongs to this browser, another live browser, or a stale origin. Same-session browsers can attach to visible active runs without automatically claiming the stream. Killing a run is now an explicit action.

Closing a running tab no longer has to mean killing the backend process. The app can ask whether to keep the command running in the background, kill it, or cancel the close.

Long-output behavior also improved:

- high-volume rendering trims rows more efficiently
- client-side output replay batches large chunks instead of appending thousands of single lines
- signal discovery skips plain output early when backend metadata is absent
- line numbers keep the true emitted-output count after transcript trimming
- partial-line tools can keep heartbeats flowing before a trailing newline arrives
- stalled streams check backend active-run state before marking a tab failed
- quiet streams can return to `RUNNING` when output resumes
- stale active runs are pruned after app/container restarts when process metadata proves the old PID is gone

These changes make repeated large scans, reloads, Docker restarts, and multi-browser sessions feel much less fragile.

## Status Monitor, History, and Output Review

Status Monitor is now one of the main app surfaces instead of a small run drawer.

It includes:

- system, resource, service, and session cards
- active-run rows with elapsed time, CPU, memory, sparkline history, attach, and kill actions
- CPU-driven pulse visuals
- command treemap, activity heatmap, recent-run constellation, and event ticker
- mobile and desktop menu access, including `Alt+M` / `Option+M`

The terminal `runs` command now reports active run IDs, PIDs, elapsed time, CPU percent, RSS memory, and command text. `jobs` aliases the same app-run view. `runs -v` and `runs --json` provide deeper details for debugging and scripting.

History gained a run-comparison flow. Users can compare completed runs against suggested or manually selected previous runs, load older candidates grouped by day, highlight changed lines, and restore both sides into fresh tabs.

Output findings are now backed by server-provided metadata. Findings, warnings, errors, summaries, and scoped navigation survive live output, history restore, share payloads, and permalink views. Built-in command output is excluded from findings counts so help/status/catalog text doesn't create noise.

History loading is page-bounded now too. The drawer and mobile recents sheet request `history_panel_limit` rows per page, and SQLite applies that limit before results are returned. Mobile recents preload their first page so the sheet opens at a stable height.

## Theme and UI Polish

The theme system has been cleaned up around shared surface roles instead of one-off component colors.

Notable changes:

- modal-like surfaces now share `modal_bg`
- shell chrome uses shared `chrome_*` roles
- dropdowns, buttons, chips, badges, form controls, drawer/sheet shells, HUD cells, and mobile handles use documented primitives
- Options and History filters now use the app-native dropdown primitive
- destructive actions are visually quieter until the actual confirmation step
- shipped themes carry the same complete token set, with drift tests guarding the contract
- retired pre-redesign tab and surface tokens were removed from built-in defaults, shipped themes, generated examples, and `THEME.md`

The day-to-day UI also picked up smaller fixes around command recall, terminal confirmations, session-token migration/revocation, path display, tab scrolling, text selection, fast-stream tail pinning, footer project naming, and Redis HUD state.

## Operator Diagnostics and Hardening

The hardening pass focused on Files, scanner execution, and diagnostics:

- workspace-rewritten command arguments are shell-quoted before crossing the `sh -c` execution boundary
- workspace reads/downloads use final-component no-follow opens where supported
- startup warns when the entrypoint workspace path and app config disagree
- workspace permission repair is safer around scanner-owned child folders
- signal-summary rendering avoids `innerHTML`
- `/diag` now auto-refreshes every 10 seconds while visible
- `/diag` Database, Redis, Vendor Assets, Config, Tools, and Top Commands cards carry deeper checks and clearer failure states
- Redis diagnostics include run-broker mode, key counts, stream lengths, memory/persistence/stats/client data, and orphan probes where available
- Database diagnostics include row counts, WAL size, reclaimable pages, query timing, and FTS orphan checks

The production Compose override now includes the documented `./workspaces:/workspaces` mount. Operators enabling Files should keep the documented ownership and mode expectations in place for bind-mounted workspace roots.

## Visual Review and Test Tooling

Demo and visual-review tooling now match the app's newer surfaces:

- demo recordings start from a fresh seeded session
- demo and capture flows exercise Files and Status Monitor
- OBS is now the standard demo recording path for desktop and mobile
- UI capture can run light-only, dark-only, or all-theme passes
- capture output includes a static review page with grouped scenes and an image viewer

The JavaScript unit harness also moved more shared behavior into small production helper seams, including autocomplete, app preferences, session identity, workspace paths, runner helpers, history formatting, search helpers, and output normalization. This keeps tests closer to the real code paths without making the app harder to reason about.

Local pytest and Playwright npm scripts now delegate to repo-owned wrappers that carry the expected config, environment, port cleanup, and quieter server-log handling.

## Compatibility and Upgrade Notes

- Files/workspace storage remains disabled unless configured by the operator.
- Deployments that enable workspaces should provide the documented writable workspace mount and permissions.
- `allowed_commands.txt` and `autocomplete.yaml` are replaced by the command registry model. Local command customization should move through `commands.local.yaml`.
- `commands.yaml` now owns command metadata for file flags, domains, wordlists, grouped short flags, runtime adaptations, examples, smoke coverage, and workflow coverage.
- The smoke-test base image now uses a stable cache tag for expensive layers, while per-run Compose projects and runtime images remain isolated.
- Project naming now consistently uses `darklab_shell` across package, container, build, footer, and docs surfaces.

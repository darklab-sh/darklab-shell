# darklab_shell v1.6

## Highlights

### Start here

If you only skim a few parts of this release, start with these:

1. **Files and session workspaces are now real app primitives** — users can create, view, edit, download, delete, recursively remove folders, browse per-session files, and use cwd-aware terminal file commands without giving commands raw shell filesystem access.
2. **Workspace-aware scanner commands are safer and more useful** — declared input/output flags for tools like `nmap`, `curl`, `ffuf`, `nuclei`, `gobuster`, `masscan`, `sslscan`, `testssl`, `nikto`, and `wpscan` can read/write guarded session files, and nmap now uses TCP connect scans by default instead of unreliable raw-socket mode.
3. **Guided workflows are user-saveable playbooks** — built-ins still provide ready-made recon sequences, and users can create, edit, delete, list, inspect, and run session-scoped workflows with `{{variables}}` from the Workflows panel or the `workflow list/show/run` command.
4. **Surfaced commands now come from one registry** — command policy, catalog metadata, ranked autocomplete, runtime adaptations, smoke examples, seeded history, wordlist value typing, and workflow command coverage now share `commands.yaml`.
5. **Output review is much stronger** — findings, warnings, errors, summaries, and scoped navigation are backed by server-provided metadata that survives live output, history restore, and share/permalink views.
6. **Long runs, reloads, and repeated scans are more resilient** — high-volume output renders faster, partial-line tools keep heartbeats flowing, disconnected run drains are bounded, stale active runs are pruned after app/container restarts, `runs` / `jobs` expose terminal active-run metadata with CPU percent / RSS-memory snapshots, SIGTERM-terminated runs no longer inflate failed-command stats, the HUD/shortcut Status Monitor shows system/resource/session cards, a right-to-left CPU-driven heartbeat strip with fresh-beat glow, eased whole-strip load-tier background color, fresh soft dashed runway reset on each monitor open, continuous no-backtracking path rendering, decoupled history-insights polling, render-cost optimizations for the visual showcase, and subheader CPU/uptime readout, ISO-aligned activity heatmap labels/popovers, adaptive 30/90-day command territory and constellation windows, full-width constellation layout, command-territory history click-throughs, constellation run restore, sparse-aware recent-run constellation popovers, event ticker, active-run rows directly under the pulse strip with wide CPU/MEM sparklines, ownership chips, circular CPU/RSS memory meters, and mobile stacked rows, mobile tabs expose active runs through the bottom peek menu, same-session browsers can attach to and explicitly kill visible active runs without an extra ownership flow, running-tab close prompts can detach the tab while leaving the backend run active, History rows can compare completed runs against suggested/manual previous runs with day-grouped older-run pagination, inline changed-line highlighting, and restore both compared runs into fresh tabs, and restored tabs return to the prompt tail.
7. **Session command variables make repeat scans easier** — `var set HOST ip.darklab.sh`, `$HOST` / `${HOST}` substitution, `var list`, and `var unset HOST` provide app-mediated per-session target reuse before normal command validation.
8. **Theme, chrome, and frontend primitives are more consistent** — modal-like surfaces share `modal_bg`, shell chrome shares `chrome_*` roles, action/selection text roles are present in every shipped theme, app-native dropdowns replace browser-native select popups in Options and History filters, and repeated controls now compose documented UI primitives instead of one-off button-like styling.

The detailed notes below summarize the release themes. `CHANGELOG.md` remains the exhaustive source for every fix and test note.

### Session Files and Workspace-Aware Commands

v1.6 adds guarded, app-mediated per-session workspace storage. Files live inside a validated session boundary with quota checks, symlink rejection, safe relative paths, cleanup by inactivity, and deployment-facing defaults that keep the feature disabled until an operator enables it.

The new Files surface supports:

- file and folder listing with breadcrumbs
- an app-native New Folder dialog with inline validation
- folder-relative New File saves
- read-only file viewing with richer previews for JSON, JSONL/NDJSON, CSV/TSV, XML, raw HTTP responses, and large line-oriented text, plus manual and auto-refresh for open previews
- edit, download, and delete actions from the viewer header
- count-aware folder deletion that warns before removing files inside a folder
- manual refresh for command-generated files
- autocomplete hints from currently loaded workspace files

The terminal also gets file-focused helpers:

- `file list`
- `file show <file>`
- `file add <file>`
- `file edit <file>`
- `file download <file>`
- confirmed `file rm <file-or-folder>`
- cwd-aware `cd`, `pwd`, `ls`, `ls -l`, `cat <file>`, `mkdir`, and confirmed `rm <file-or-folder>` aliases when Files are enabled

Each tab now keeps its own workspace current directory, with the workspace root displayed as `/`. File commands resolve relative paths from that directory, short listings are non-recursive by default, long listings align columns, and directory deletion requires recursive flags such as `rm -r` or `rm -rf`. Short `ls` / `file list` output stays compact when printed directly, but pipe helpers receive one logical entry per line so `ls | grep txt` behaves like a normal terminal. The desktop prompt now combines the configured identity prefix with the current path, so workspace-enabled tabs show prompts like `anon@darklab:/reports $` while workspace-disabled sessions keep the familiar `anon@darklab:~ $` shape. Mobile keeps that path context in the empty command placeholder instead, using compact text such as `/reports · type command` or `.../nuclei · type command`.

The Files viewer now opens in its own modal and previews common scanner output shapes without changing the underlying file. JSON/XML get formatted line previews, JSONL/NDJSON files parse and pretty-print each record, CSV/TSV render as scrollable tables with sticky headers, raw HTTP responses separate status line, headers, and body, and large text files show a line-numbered bounded first chunk of up to 10,000 lines with large-preview search guards based on line count, character count, or file size, debounced terminal-style search, lazy current-match highlighting, case sensitivity, regex, previous/next controls, and Enter-to-next behavior outside the scrollable preview. The viewer opens immediately with loading feedback while larger files are read and formatted, and mode switches or manual refreshes use the same feedback before re-rendering. Open previews can be refreshed manually or opt into auto-refresh every five seconds, with bottom-follow behavior only when the user is already at the end of the file. Auto-refresh is unavailable for files larger than 1 MB to avoid repeated formatting work while browsing large outputs. New and edit file flows also open in their own modal instead of inline inside the Files browser. Raw view, edit, download, and delete stay available for every preview.

Scanner examples can now declare workspace-required behavior. File input/output flags are rewritten into the guarded workspace only when they are explicitly declared, so useful command shapes become possible without opening broad shell navigation, redirection, or arbitrary path access. ProjectDiscovery tools also surface selected secondary outputs through the same model: `katana` response/field directories, `pd-httpx` response/screenshot directories, `nuclei` response stores/exports/logs, `subfinder` per-domain directories and config reads, and `naabu` auxiliary input files can all point at session-visible Files paths while provider/API-key configs remain session-owned rather than packaged into share snapshots.

The production Compose override now includes the documented `./workspaces:/workspaces` mount. Operators should keep the documented ownership and mode expectations in place for bind-mounted workspace roots.

### Command Registry, Workflows, and Smoke Coverage

The old split between allowlist data, autocomplete data, seeded examples, smoke examples, and workflow command lists created too many chances for drift. v1.6 moves the surfaced external command model into `app/conf/commands.yaml`.

That registry now feeds:

- command policy and catalog metadata
- autocomplete hints and examples
- command-specific runtime adaptations such as safe default flags, command-prefix wrappers, managed workspace directories, and environment wrappers
- smoke-test command corpus
- seeded history
- workflow command coverage
- workspace-aware file flag declarations
- explicit value metadata such as `value_type: domain`

Workflows also become more useful in v1.6. Guided workflows can declare structured inputs, render commands from those values, preserve input state while the modal is open, and run all steps sequentially in the active tab. Users can now save session-scoped workflows from the Workflows panel, edit/delete them above the built-in catalog, and run them from the prompt with `workflow list`, `workflow show <name>`, and `workflow run <name> --input value`; missing required inputs are prompted in the transcript. Two Files-backed examples, `Subdomain HTTP Triage` and `Crawl And Scan`, demonstrate app-native recon chaining by writing generated session files and feeding them into follow-up tools through declared workspace-aware flags.

Built-in autocomplete now uses the same registry-shaped grammar without turning built-ins into operator config. App-owned grammar lives in `app/builtin_autocomplete.yaml`, `/autocomplete` merges it with external command metadata, and the browser only layers dynamic values such as workspace files, session variables, theme names, config options, and command lookup targets on top.

Autocomplete matching now ranks exact, prefix, token-boundary, substring, and fuzzy hits across command roots, subcommands, flags, arguments, workspace files, recent domains, and wordlists. Workspace file paths and installed SecLists paths can match useful path segments or filename fragments, while contextual value-slot rules still decide when domain, wordlist, or workspace-file candidates are eligible. Suggestion rows highlight matched spans or fuzzy characters, and curated command examples keep their authored order once matched.

Domain autocomplete is now metadata-driven as well. Flags and positional arguments that expect domains can declare `value_type: domain`, and the browser uses that explicit marker to capture up to 10 recent domains per active session token and suggest them only in matching domain slots. Recent domains are stored server-side with the session, so they follow named tokens across browser restarts, devices, and private windows instead of disappearing with browser `sessionStorage`.

SecLists discovery is now catalog-backed. The built-in `wordlist` command lists curated installed categories, searches entries, prints copy-friendly paths, and can opt into `wordlist --all` for the full scanned tree. Command metadata can mark values with `value_type: wordlist` plus `wordlist_category`, so autocomplete offers installed SecLists paths only in relevant slots while preserving session-file suggestions for workspace-aware flags.

Operators can also configure `restricted_command_input_cidrs` to block literal IP/CIDR inputs in declared target slots. The validation path uses the same `commands.yaml` value metadata, catches IPs embedded in URLs or host:port values, and inspects app-managed workspace target-list files when a declared read flag points at one. Domain names are intentionally not DNS-resolved by this first pass.

Container smoke tests now cover both command-registry examples and workflow commands. The workspace-enabled smoke path also creates session files, runs representative file-read/file-write commands, verifies generated output, and cleans up through the Files API.

### Output Findings and Review

Output review now has a server-owned signal contract. Instead of relying on browser-only DOM rescans, `/runs` classifies streamed output lines and persists signal metadata through history, full-output artifacts, share payloads, and permalinks.

The UI uses that metadata for:

- findings, warnings, errors, and summaries counts
- scoped signal navigation
- search discoverability on tabs with findings
- transcript-native `Command Findings:` summaries
- per-command and per-target grouping where metadata is available

Built-in command output is excluded from signal counts so help/status/catalog text does not create noisy findings.

### Long Output, Stalls, and Reload Recovery

Several reliability fixes landed around long-running commands and large transcripts:

- normal command starts now use a backend-owned `/runs` stream foundation, with Redis-backed replayable events for production deployments, a single-process fallback for local development, and active-run reload recovery that subscribes back to available live output
- high-volume rendering now trims rows more efficiently and avoids expensive renumbering work after the live line cap
- repeated workspace reads now keep signal discovery incremental: plain output without backend signal metadata is skipped early, search affordance counts use per-tab signal caches, and client-side multi-line output replays through chunked batches instead of thousands of single-line DOM-facing appends
- line numbers preserve emitted-output order after transcript trimming, so the prompt shows the true next output line even when old rows have been dropped
- partial-line tools can keep SSE heartbeats flowing without waiting for a trailing newline
- disconnected run drains are bounded to the configured command timeout plus grace so orphaned scanners cannot pin active-run state indefinitely
- stalled live streams check backend active-run state before marking a tab failed
- if a quiet stream resumes, the UI returns the run to `RUNNING` and keeps output flowing in place
- active-run metadata records PID start time when Linux `/proc` is available, so reload reconnect can prune commands that died with an old container
- stalled-run recovery uses a run-generation guard so older timeout checks cannot overwrite newer tab state after rapid tab switches, kills, or restarts
- restored tabs re-stick to the prompt tail across delayed layout ticks, prompt mount, and activation of previously hidden long-output tabs

Together these changes make big scanner runs feel less fragile, especially around reloads and Docker/app restarts.

### Status Monitor and Active-Run Visibility

v1.6 adds a real app-native active-status monitor instead of relying on shell-job-shaped approximations.

The terminal `runs` command now reports active run IDs, PIDs, elapsed time, CPU percent, RSS memory, and command text. `jobs` aliases the same app-run view. `runs -v` exposes full IDs, started timestamps, cumulative CPU time, RSS memory, and metadata source, while `runs --json` returns the active-run snapshot as JSON for scripting or debugging.

The desktop rail, `STATUS`, `LAST EXIT`, and `TABS` HUD cells now open the same monitor, and `Alt+M` / `Option+M` opens it from the keyboard. The desktop surface is a centered modal with backdrop/focus/dismiss behavior, uses the shell chrome theme roles, lists active commands as divided rows with a green accent, and switches to a run's tab when that row is clicked. If nothing is running, the monitor stays useful as a `0 active runs` state.

Resource telemetry is best-effort and fail-soft. The backend samples the tracked process plus recursive children through `psutil` when available, reports cumulative CPU seconds and RSS bytes, and the browser calculates live CPU percentages from adjacent poll samples. The UI caps displayed CPU at 100% and renders CPU/RSS as circular meters, with memory normalized against a 1 GB meter scale while still labeling the actual memory value. Because CPU percent needs two samples, the monitor now primes a background sample pair when runs start, keeps a slow closed-state cache while visible, renders first-sample CPU as a 75%-filled rotating meter, and does a quick follow-up refresh when opened before a percent is ready.

Active-run recovery now distinguishes session identity from browser origin. Each browser has a local client id, `/runs` records the originating client/tab, and `/history/active` reports whether an active run belongs to this browser, another live browser, or a stale origin. A phone or second browser using the same session token can monitor the command without automatically creating a terminal tab or claiming the stream. Status Monitor exposes explicit Attach and Kill actions for visible current-session runs; Attach opens a subscribed tab with replay plus live output, and Kill intentionally terminates the backend process group for every subscriber. Closing a running tab now asks whether to keep the command running in the background, kill it, or cancel.

When many tabs have live broker streams open, the browser temporarily pauses non-active stream subscriptions while Status Monitor is open so its `/status`, `/history/active`, `/history/insights`, and workspace-status requests are not queued behind long-lived SSE connections. Paused tabs resume from their last broker event id when the monitor closes, and active-run polling refreshes same-client owner liveness while those streams are paused.

The Command Constellation also got a polish pass. Synthetic app built-ins (`pwd`, `whoami`, `help`, …) are filtered server-side from the constellation, treemap, heatmap, events, and `max_day_count` so the visualizations show real recon work only. The Y axis now crops at the p98 of `elapsed_seconds` plus a small headroom pad, quantises to 10% of the full range, and floors at 25% so the bulk of stars climb off the baseline even when one multi-hour scan would otherwise flatten the canvas — stars above the ceiling clamp to the top edge with an upward tick. Streak connectors require same-root, ≤2 hours between consecutive starts, and the same calendar date, so a session at 23:30 → 00:30 splits at midnight rather than wrapping back across the canvas. The misleading `24` axis label has been dropped (the X mapping caps at 23:59), and the legend now keys both the red failure ring and the small/large output-size encoding. The Y axis carries five log-scale tick labels (`0s` through the dynamic ceiling) plus matching horizontal major/minor guide lines, so a star's elapsed time can be read off the canvas instead of inferred from the popover. `/history/insights` is now event-driven: it loads once on first open and refreshes exactly once when the active-run count transitions `>0 → 0`, so a freshly completed run lands in the heatmap "today" cell, the treemap percentages, and the constellation without a polling timer.

Mobile gets the same active-run affordance in a smaller shape. The normal mobile menu includes Status Monitor so idle users can attach to or kill another browser's current-session run, while the active-tab bottom peek still switches from Recents to the Status Monitor with live elapsed time during a command. After the run completes, the peek returns to Recents. The first run in the browser session nudges the peek upward for discoverability and respects reduced-motion preferences.

History loading is also page-bounded now. The drawer and mobile recents sheet request `history_panel_limit` rows per page, and the backend applies that limit in SQLite instead of materializing the full session history before slicing. Mobile recents preloads the default first page and shows a stable loading row when opened before the cache is ready, so the sheet no longer grows from an empty state after history data arrives.

### Hardening and Operator Diagnostics

The v1.6 hardening pass tightened several internals that matter most once Files and long-running scanners are enabled:

- workspace-rewritten command arguments are shell-quoted before crossing the existing `sh -c` execution boundary
- workspace reads and downloads use final-component no-follow opens where supported, closing a theoretical symlink-swap gap after validation
- startup warns when the Docker `WORKSPACE_ROOT` path prepared by the entrypoint differs from the app `workspace_root` config
- workspace permission repair now covers files directly inside session directories, warning-logs chmod repair failures, and no longer recursively resets command-owned session folders to `appuser` before scanner-owned child repair
- frontend signal-summary rendering no longer depends on `innerHTML` for generated count markup
- `scripts/benchmark_output_signals.py` provides a manual 10 MB synthetic-output baseline for the server-side signal classifier without putting timing assertions in CI

### Theme Tokens and Frontend Primitives

The theme system received a broad consistency pass. Modal-like surfaces now share `modal_bg`, including FAQ, Options, Files, Workflows, Status Monitor, Keyboard Shortcuts, compact confirmations, restore prompts, and the Theme selector. Shell chrome surfaces now share `chrome_bg`, `chrome_header_bg`, `chrome_row_bg`, `chrome_row_hover_bg`, `chrome_control_bg`, `chrome_control_border`, `chrome_divider_color`, and `chrome_shadow`, covering the desktop rail, HUD, History drawer, Status Monitor rows, mobile recents, and mobile menu. Theme preview cards were refreshed to show a compact v1.6 shell schematic with rail sections, tabbar, panel, HUD bar, and a small modal surface instead of the old terminal-window approximation.

Lightweight theme visual drift guardrails now backstop that cleanup. The test suite checks that shipped theme YAML files carry the same complete token set, retired one-off tokens stay removed, `THEME.md`'s key reference matches runtime export order/defaults, CSS does not reference undefined `--theme-*` variables without fallbacks, new color literals stay tokenized or variable-derived, and every shipped theme renders a populated preview card for core terminal, chrome, modal, button, and dropdown surfaces.

The stale UI cleanup also retired unused pre-redesign tab theme keys from the public theme contract. `tabs_bar_scrollbar_*`, `tab_bg`, `tab_border`, `tab_active_border`, `tab_active_text`, and `tab_active_shadow` are gone from built-in defaults, shipped themes, generated examples, and `THEME.md`; live tab styling now keeps only the tokens the current tab strip actually consumes.

Several one-off or legacy roles were retired in favor of canonical keys: old per-surface modal backgrounds, confirm modal backgrounds, History drawer background/shadow roles, mobile menu roles, tab-scroll button roles, chip roles, dropdown-up roles, and form-control one-offs. FAQ code and allowed-command chips now share `inline_surface_bg`, and all shipped themes now include action/selection text roles:

- `on_accent_text`
- `selection_text`
- `selection_line_text`
- `modal_danger_btn_text`
- `modal_warning_btn_text`

Options and History filter selects also now use the shared app-native dropdown primitive. The original `<select>` remains in the DOM as the state owner and accessibility/test fallback, while the visible control renders a themed button/listbox menu with synchronized `change`, outside-click, Escape, and keyboard behavior.

The frontend primitive pass now covers buttons, nav rows, close controls, toggles, keyboard helper keys, dropdown items, chrome rows, panel rows, chips, badges, form controls, drawer/sheet shells, HUD action cells, and mobile sheet gesture handles. The button primitive allowlist is intentionally empty of exceptions, so future button-like surfaces need to pick a documented primitive or add a reviewed exception.

Destructive entry points were also toned down. History/Recents `delete all` and row `delete`, Files `Delete`, and Options `Clear` now use neutral secondary buttons in their initial surfaces. Their confirmation modals keep the destructive button styling, so the red visual weight appears at the actual point of confirmation rather than across normal browsing surfaces.

The current theme registry reflects the shipped 18-theme set, generated light/dark theme examples follow the new canonical sections, legacy preview-only window-button keys are gone, and theme preview cards no longer show old traffic-light terminal-window chrome.

### UI and Operator Polish

v1.6 includes several smaller fixes that remove rough edges from daily use:

- session-token revocation now requires transcript confirmation and warns that token-backed history and workspace files will not be recoverable through the app after revocation
- session identity migration now carries workspace files and empty folders forward when accepting a token migration, while counting skipped conflicts/quota misses
- command recall is tab-local first and then falls back to deduped session-wide history, so empty/new tabs can still recall previous commands without trapping each tab in its own short history
- built-in command output now uses shared ANSI helpers for underlined headers, muted metadata, green online/active/success states, amber disabled states, and red failures across `runs`, `last`, `status`, `stats`, `limits`, `retention`, `file list`, `ps`, `route`, `df -h`, and `free -h`
- terminal yes/no prompts ignore command history navigation and autocomplete while a confirmation is pending
- the Redis HUD pill clears stale `ONLINE` state when the server becomes unreachable
- Files view opens at the top of each file instead of preserving a stale bottom scroll position
- footer/version project links consistently use `darklab_shell`
- tab-scroll glyphs no longer flash on reload before overflow state settles
- desktop prompt text selection behaves more like normal transcript selection
- fast streaming output stays pinned to the live tail unless the user actually scrolls away
- recent-domain autocomplete uses explicit `commands.yaml` `value_type: domain` slots instead of placeholder-name guessing
- grouped short-flag policy uses explicit `commands.yaml` `allow_grouping` metadata, so `nc` can allow `-zv` / `-vz` / `-zvn` without listing every ordering

The docs drift tests now also enforce stable appendix and README project-tree ordering so future doc updates keep following the tracked file list.

### Visual Review and Demo Tooling

The visual tooling received a practical cleanup pass:

- demo recordings start from a fresh seeded session by default
- the `visual-flows` seed keeps Recent and starred rows readable
- UI capture can run light-only, dark-only, or all-theme passes
- capture runs generate a static `index.html` review page with grouped scenes and an image viewer
- demo and capture flows now exercise Files by generating and opening a workspace file
- the desktop demo and capture pack now open the Status Monitor during an active command and pause after CPU/RSS memory telemetry has populated
- repeated dropdown, scrollbar, search-highlight, signal-search, toast-shadow, and shared button primitive color transforms now resolve through derived theme tokens in every shipped theme instead of duplicated component-local `color-mix()` formulas
- local pytest and Playwright npm scripts now delegate to repo-owned wrappers that carry the expected config, environment, and port-cleanup defaults

These are mostly release-engineering improvements, but they make visual review faster and reduce stale-session artifacts in generated media.

### Compatibility and Upgrade Notes

- Files/workspace storage remains disabled unless configured by the operator.
- Deployments that enable workspaces should provide the documented writable workspace mount and permissions.
- `allowed_commands.txt` and `autocomplete.yaml` are replaced by the command registry model; local command customization should move through `commands.local.yaml`.
- The smoke-test base image now uses a stable cache tag for expensive layers, while per-run Compose projects and runtime images remain isolated.
- Project naming now consistently uses `darklab_shell` across package/container/build surfaces.

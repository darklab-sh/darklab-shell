# Changelog

All notable changes to darklab shell are documented here.

---

## [1.4] ŌĆö 2026-04-11

### Fixed
- **Firefox mobile keyboard shell instability** ŌĆö the mobile terminal shell was rebuilt around a simpler normal-flow layout after the investigation proved the browser itself was not the root cause. The fixed-shell / page-scroll / `visualViewport` pan-compensation stack was removed, `#mobile-shell` now stays in normal document flow, and the composer block stays layout-driven instead of relying on page transforms, scroll correction, or keyboard-height spacing on `#mobile-composer-host`. This removes the intermittent flash, gray-gap, floating-composer, and repeated open/close drift regressions that were showing up in Firefox mobile when the keyboard opened.
- **Mobile keyboard open/close consistency** ŌĆö the mobile keyboard helper row and composer now follow the focused visible mobile input without synthetic tap-to-focus handlers on the input or lower composer area. The keyboard-open path keeps the last real open offset when viewport reporting lands late, stores a closed-height baseline for later comparisons, and resets the closed layout immediately on hide / blur / backgrounding so repeated opens do not accumulate position drift.
- **Mobile shell polish** ŌĆö the mobile transcript/composer boundary now keeps a consistent gap in both closed and keyboard-open states, the permalink/copy/save action row uses the same horizontal gutter as the mobile composer block, and the mobile header status pill was re-aligned next to the hamburger button for a cleaner stacked layout.
- **Recent-command chip row alignment and overflow** ŌĆö the recent-command strip now sits closer to the header instead of floating inside the global page gap, the chip contents are optically centered between the two seams, and the desktop overflow logic now keeps the `+ more` chip on the first row instead of letting it wrap into a clipped second line.
- **Desktop and mobile tab-row seam polish** ŌĆö the tab rows now sit flush against the terminal seam without the earlier duplicate borders, gap artifacts, or transient right-edge scrollbar strip. The desktop row also hides the horizontal scrollbar chrome while preserving scrolling behavior.
- **Tab close hover tone** ŌĆö the tab close button no longer uses a red destructive-looking hover ring in the dark defaults and `darklab_obsidian`. The hover treatment now follows the shellŌĆÖs neutral/accent language instead of reading like an error state.
- **Theme parity on mobile and dark-theme palette polish** ŌĆö the mobile composer shell now uses the same injected theme variables as desktop instead of hardcoded dark surfaces, and the obsidian-family themes now ship with darker base backgrounds plus lighter gray chrome values for better contrast. The built-in dark fallback palette and `theme_dark.yaml.example` were aligned with the revised `darklab_obsidian` values.
- **Mobile output follow-after-keyboard-open** ŌĆö when the mobile keyboard opens, the active tabŌĆÖs output is now re-stuck to the bottom so the last line stays visible instead of being clipped behind the composer. A focused Vitest regression locks that behavior in alongside the existing mobile-shell layout guard.
- **Live output tail helper** ŌĆö when a tab has output but is scrolled off the live tail, the output pane now shows a small jump-to-live / jump-to-bottom button that follows the tab's streaming state. The button stays tab-scoped, swaps from the live label to the bottom label when the stream ends, and snaps the output back to the live tail when clicked. A CSS `[hidden]` override now guarantees the helper actually disappears when hidden, so the button no longer looks stuck visible just because its base display rule is `inline-flex`.
- **Desktop output-selection shortcut replay** ŌĆö when output text is highlighted on desktop, pressing `ArrowUp`, `ArrowDown`, `Enter`, or `Ctrl+R` now hands control back to the prompt without dropping the shortcut keystroke. History navigation, submit, and reverse-search therefore still work immediately after selecting transcript text.
- **Regression coverage expansion** ŌĆö added a focused pytest regression for the Container Smoke Test early-kill path, plus Playwright coverage for the live output tail helper and desktop transcript-selection shortcut replay. That closes the biggest remaining gap between the user-visible browser behavior and the earlier jsdom-only coverage.
- **Content view logging expansion** ŌĆö the main content/config GET routes now emit `CONTENT_VIEWED` info logs with the client IP, session ID, and route/context fields, the home pageŌĆÖs `PAGE_LOAD` log includes the resolved theme, and theme resolution emits a `THEME_SELECTED` DEBUG breadcrumb with the theme name and source so startup and content reads are easier to correlate in production logs.
- **History list access logging** ŌĆö opening the `/history` list now emits a `HISTORY_VIEWED` info log with the session and returned run count, matching the existing permalink/share access logs and giving operators a trace for history-panel reads.
- **Composer helper state drive** ŌĆö the shared cursor and mobile edit-bar helpers now read and write selection from `composerState` instead of depending on stale DOM selection, so caret motion stays in sync with the active input even as the input-decoupling refactor continues. A targeted Vitest regression proves caret movement follows shared state rather than whatever selection the hidden desktop input last held.
- **Composer keydown / submit state drive** ŌĆö the remaining controller key handlers and submit paths now read the active composer text from shared state before they mutate, submit, or inspect it, so `controller.js` and `runner.js` no longer depend on `cmdInput.value` as the global source of truth for Enter and shortcut handling. The runner harness and app tests now seed the shared composer path directly in the phase-5 coverage.
- **Composer draft / history state drive** ŌĆö tab draft capture, hist-search restore, and the remaining chip/autocomplete restore paths now use the shared composer state instead of pulling directly from whichever DOM input happened to fire last. That keeps draft persistence aligned across desktop typing, mobile typing, and tab switches while the input-decoupling refactor continues.
- **Composer prompt render state drive** ŌĆö the shell prompt render now reads value and selection from `composerState` instead of from the hidden desktop input, so the prompt line stays correct regardless of which composer view is active. A dedicated Vitest regression proves the prompt render ignores stale DOM input state and follows the shared store.
- **Empty mobile Run button submits** ŌĆö the shared run-button sync logic now disables both desktop and mobile Run buttons whenever the active tab is running or the current composer value is empty / whitespace-only. Programmatic composer updates use the same path, so recent-command chips and other helpers cannot leave the mobile Run button enabled against a blank prompt.
- **Recent mobile command chip clutter** ŌĆö mobile recent-command chips no longer prepend the inline star glyph inside the chip body, and chip clicks blur the tapped chip before loading its command so the touch path does not keep stale focus styling around the composer.
- **Loopback address blocking** ŌĆö commands containing `localhost`, `127.0.0.1`, `0.0.0.0`, or `[::1]` anywhere in the command string are now blocked by `_LOOPBACK_RE` in `commands.py`, using the same pattern as the existing `/data` and `/tmp` path checks. This prevents web shell users from curling internal Flask endpoints (e.g. `curl http://localhost:8888/diag` or `curl 127.0.0.1:8888/config`) regardless of how the URL is formatted. Word-boundary anchors prevent false positives on external hostnames like `notlocalhost.com`. Client-side deny entries (`!curl localhost`, `!curl 127.0.0.1`, `!wget localhost`, etc.) in `conf/allowed_commands.txt` provide immediate browser feedback for the most obvious bare-hostname patterns.
- **History drawer closes on typing** ŌĆö typing in the command input while the run history drawer was open no longer left the drawer visible. The `input` event handler in `controller.js` now calls `hideHistoryPanel()` when the drawer is open, matching the behavior of submitting a command or opening a modal.
- **Modal overlay keyboard blocking** ŌĆö the FAQ, Options, and Theme modals now block all keyboard input from reaching the terminal, not just printable characters. Previously, pressing `Enter` while a content modal was open appended a new prompt line to the terminal output, and `Ctrl+C` could trigger kill-confirmation on a running tab. Both the `cmdInput` keydown handler and the `document` keydown handler in `controller.js` now return early for all keys (except `Escape`) when any of those three overlays are open. `Escape` still closes the active overlay and refocuses the terminal input.
- **App name font consistency across pages** ŌĆö the app name in the header of permalink and diag pages was using `var(--font-display)` (Syne) at 16px glow, while the main shell uses `var(--font-mono)` (JetBrains Mono) at 20px glow. The `.permalink-title` rule in `styles.css` now uses `var(--font-mono)`, 20px glow, and the same weight and letter-spacing as the main header so the app name renders identically across all pages.
- **Permalink and diag action button hover** ŌĆö the `ŌåÉ shell` link and other action buttons on permalink and diag pages had no hover highlight. A scoped `.permalink-actions .term-action-btn:hover` rule was added to `styles.css` to apply the same green border and text color used by action buttons on the main shell.
- **Gunicorn orphan-process crash (`init: true`)** ŌĆö added `init: true` to both `docker-compose.yml` and `examples/docker-compose.standalone.yml` so Docker's bundled tini process runs as PID 1. Without this, scanner tools like `wpscan` that exit with code 3 as a normal status code (meaning "potentially interesting findings") could crash the server: when the entire subprocess chain receives SIGTERM simultaneously, intermediate parents may die before the leaf tool, orphaning it to PID 1 (Gunicorn). Gunicorn's `reap_workers()` collects the orphan via `waitpid(-1)` and interprets `exit(3)` as `WORKER_BOOT_ERROR`, halting the server. With `init: true`, tini is PID 1, Gunicorn is PID 2+, and orphaned scanner processes are silently reaped by tini.
- **`/kill` endpoint PGID race condition** ŌĆö the endpoint now uses the stored subprocess PID directly as the PGID instead of calling `os.getpgid(pid)` at kill time. Since all subprocesses are spawned with `preexec_fn=os.setsid`, PGID equals PID at creation, so the stored value is always correct. The previous `os.getpgid()` call would return the wrong PGID if the subprocess had already been reaped and its PID reused by a new Gunicorn worker, causing `kill -TERM -<worker_pgid>` to send SIGTERM to the entire Gunicorn worker pool.
- **Late kill / exit prompt race** ŌĆö when a command finishes at the same time the user clicks Kill, the runner now re-attempts deferred prompt mounting after the kill path so the active tab does not get stuck without the prompt below the output. The same prompt-remount path also runs when the late exit event arrives after a kill click.
- **Container Smoke Test DinD filesystem mismatch** ŌĆö the `container_smoke_test` fixture no longer depends on runtime bind mounts that assume the Docker daemon can see the test process's filesystem. It now builds the base image, creates a temporary container, copies the repo `app/` tree plus a generated `config.local.yaml` into `/app`, commits that as a runtime image, and starts compose from that image with no client-side bind mounts. This fixes the GitLab DinD case where the daemon runs in a separate sidecar and cannot reliably mount paths from the job container.
- **Container Smoke Test DinD port selection** ŌĆö the fixture no longer guesses a free localhost port from the job container namespace. It now publishes `8888` on an ephemeral host port and discovers the real published port with `docker compose port`, which avoids cross-namespace port collisions under DinD.
- **Container Smoke Test corpus refresh** ŌĆö the smoke-test expectation baseline was refreshed for current `host`, `dig`, `nc`, and `dnsrecon` output so the fixture keeps matching the upgraded tool behavior in `app/conf/auto_complete.txt`.
- **SQLite bootstrap race under Gunicorn startup** ŌĆö `db_init()` now takes a filesystem lock before schema creation, migration, and retention pruning. Without that lock, multiple Gunicorn workers importing `database.py` at the same time could race through `PRAGMA journal_mode=WAL` and startup migration work, intermittently failing with `sqlite3.OperationalError: database is locked`.
- **Search bar hidden-on-load** ŌĆö `ui_helpers.js` now sets `searchBar.style.display = 'none'` at initialisation so the inline style overrides the conflicting `.search-bar { display: flex }` rule (same specificity, later in the stylesheet) that was causing the bar to appear visible on load when `.u-hidden` was also applied. `isSearchBarOpen()` was also tightened from `!== 'none'` to `=== 'flex'` so the bar is treated as closed before any JS interaction sets the inline style explicitly.
- **Blank Enter while a command is running** ŌĆö pressing Enter on an empty prompt while a tab is in the `running` state no longer appended a spurious blue prompt line to the output. The empty-command path in `submitCommand` now returns early when the active tab is running.
- **Deny-prefix flag matching rewrite** ŌĆö `_is_denied()` now tokenizes both the command and deny entries using the shared `split_command_argv` helper, matches tool and subcommand prefixes case-insensitively, and matches flags with exact case. Previously, the entire deny entry was lowercased before comparison, so `!curl -K` also blocked `curl -k`. The new implementation also handles combined short flags (e.g. `-sU` is caught by a `!nmap -sU` deny entry) correctly.
- **Draft input corrupted when switching tabs with a keyboard shortcut** ŌĆö pressing a keyboard shortcut that contains the Tab key (such as `Alt+Tab`) while `cmdInput` had focus triggered the autocomplete accept handler (`acAccept`) because the `Tab` key case in the `cmdInput` keydown handler had no modifier guard. If `acFiltered` still held stale suggestions from the previous tab's typing session (e.g. `['whoami']`), `acAccept` would overwrite the current input and fire an input event that saved the wrong value into the active tab's `draftInput`. On the next switch, that tab's draft was restored as the stale suggestion instead of the original command. Fixed by adding `!e.altKey && !e.ctrlKey && !e.metaKey` guards to the `Tab` handler in `controller.js` so modifier-key combinations no longer trigger autocomplete logic.
- **Stale autocomplete suggestions persisting across tab switches** ŌĆö `activateTab` in `tabs.js` now calls `acHide()` and resets `acFiltered = []` when switching tabs, so suggestions filtered for one tab's typing session cannot bleed into another tab. Without this, pressing bare Tab on a newly activated tab could accept a suggestion from the previous tab's last input instead of filtering for the current draft.
- **Ctrl+R hist-search auto-fill** ŌĆö typing in hist-search mode immediately overwrote the input with the first matched command, so continuing to type appended to the full match instead of refining the query (e.g. typing `w` matched `whoami` and populated the field, so typing `ho` then produced `whoamiho`). `handleHistSearchInput` no longer calls `setComposerValue` during typing; the typed query stays in the input while matches appear in the dropdown. ArrowDown/Up and Ctrl+R navigate the list and fill the input with the highlighted match; Tab accepts without running; Enter accepts and runs.
- **Ctrl+C in hist-search left autocomplete open** ŌĆö pressing Ctrl+C to exit hist-search called `exitHistSearch(false)`, which restored the pre-search draft via `setComposerValue` with `dispatch: true`. That fired an input event that triggered `handleComposerInputChange`, reopening the autocomplete dropdown if `acFiltered` had entries. A second Ctrl+C was required to dismiss it. Fixed: `exitHistSearch` now accepts a `keepCurrent` option. The Ctrl+C handler passes `{ keepCurrent: true }` so the typed query stays in the input instead of restoring the pre-search draft. `exitHistSearch` also calls `acHide()` unconditionally at the end so autocomplete cannot reopen regardless of the exit path.
- **Hist-search ArrowDown/Up fell through to command history navigation** ŌĆö pressing ArrowDown/Up while hist-search was active was not handled by `handleHistSearchKey`, so the keys fell through to the command history navigator instead of navigating the hist-search dropdown. Added ArrowDown/Up cases that step `_histSearchIndex` through the match list, fill the input with the selected match via `setComposerValue`, and re-render the dropdown highlight.
- **Hist-search Tab accepted from the wrong source** ŌĆö Tab while hist-search was active fell through to the `acFiltered`-based autocomplete Tab handler in `controller.js`, which could accept a stale suggestion unrelated to the hist-search results. Added a Tab case to `handleHistSearchKey` that calls `exitHistSearch(true)` to accept the currently selected hist-search match and close the dropdown without running the command.
- **Hist-search Enter did not run the command** ŌĆö `exitHistSearch(true)` on Enter filled the input with the selected match but never called `submitComposerCommand`, so the menu closed without executing anything. The Enter handler now calls `submitComposerCommand(cmdInput.value, { dismissKeyboard: true })` after accepting. When there are no matches it exits with `keepCurrent: true` (preserving the typed query) and runs that instead of restoring the pre-search draft.
- **Hist-search typing appended to pre-search draft** ŌĆö `enterHistSearch()` did not clear the input when activating search mode, so if the user had a command like `ping -c 10 google.com` in the input and pressed Ctrl+R, the first keystroke would append to that text rather than starting a fresh query. Fixed: `enterHistSearch` now saves the current input as `_histSearchPreDraft` and then clears it (without dispatching) so subsequent typing goes directly into the search query. Escape and Ctrl+G still restore the original draft.
- **Mobile edit-bar hold-to-repeat** ŌĆö the left and right cursor-move buttons in the mobile composer now repeat while held: a 400 ms initial delay starts the repeat, which then fires every 60 ms. Document-level `pointerup`, `pointercancel`, `touchend`, and `touchcancel` safety listeners ensure the timer is always cancelled even if the pointer leaves the button while held. Previously a single tap moved the cursor one position with no repeat.
- **Mobile composer tap-to-reposition cursor** ŌĆö tapping on the mobile composer input to reposition the cursor within the existing text no longer snaps back to the end of the line. The `pointerdown` and `touchstart` handlers now return early (without calling `preventDefault` or `focus`) when the input is already focused, so the browser can place the cursor at the tapped position normally. A `click` handler additionally captures the tapped cursor position and restores it on the next tick if a downstream synchronisation call moves it to end.
- **Mobile composer cursor-to-end jitter** ŌĆö after tapping to reposition, the cursor was intermittently jumping back to the end of the line. The root cause was a cascade: `setComposerValue` dispatched an `input` event on the desktop input, which re-read the desktop `selectionStart` (still at end) and wrote that back to the mobile input via `setSelectionRange`. Fixed by adding an `exclude` option to `setComposerValue` that skips the `.value` assignment on the source input when called from `handleComposerInputChange`, and by rewriting `syncHiddenCommandInputFromVisible` to write directly to `cmdInput` without triggering the dispatch cascade.
- **Mobile `composerRefs.row` stealing focus from taps** ŌĆö tapping on the mobile composer input bubbled to the row's `pointerdown`/`touchstart` handlers, which called `focusCommandInputFromGesture()` and reset the cursor. The row handlers now exclude the mobile input element as a valid trigger target.
- **macOS accent picker / hold-to-repeat on desktop** ŌĆö holding a key on the desktop prompt on macOS would open the OS accent chooser instead of repeating the character. The `cmdInput` keydown handler now calls `e.preventDefault()` for all printable, non-modifier, non-IME keys and manually inserts the character via `replaceCmdRange`, signalling to the OS that the key event is handled and suppressing the accent picker. The handler is skipped while the welcome intro is active so Space/Enter/Escape welcome-settle triggers and printable insertion remain under the `document` keydown handler that owns welcome key routing.
- **Desktop hidden input soft-keyboard guard** ŌĆö `#cmd` (the hidden desktop input) now carries `inputmode="none"` so mobile browsers do not raise the soft keyboard if the element receives focus unexpectedly.
- **Desktop command chip row overflow** ŌĆö the recent-command chip row now truncates to a single line on desktop. Chips that would wrap to a second row are removed and replaced with a `+ more` chip that opens the run history drawer, matching the behaviour of the mobile chip row (which caps at three chips). The chip count re-evaluates on window resize. A `max-height` clip on `.history-row` ensures any briefly-wrapped chips during render are never painted before JS removes them.
- **Text selection in terminal output** ŌĆö clicking and dragging to select output text no longer immediately clears the selection by stealing focus back to the command prompt. The `terminalBody` click handler now skips `focusAnyComposerInput` when `window.getSelection().toString()` is non-empty so a just-completed drag selection survives.
- **Keystrokes forwarded after selecting text** ŌĆö pressing a printable key while focus is outside the command input (e.g. after selecting output text) now forwards the keystroke to the prompt instead of discarding it. A catch-all at the end of the `document` keydown handler focuses `cmdInput`, inserts the character via `replaceCmdRange`, and calls `e.preventDefault()`. The catch-all is skipped when a modal is open, when the target is a button or link, or when the target is an editable field.
- **Scroll to bottom on typing** ŌĆö typing in either the desktop or mobile composer now immediately scrolls the active output to the bottom and resets `followOutput` to `true`. Previously, if the user had scrolled up to review output, typing did not return the view to the prompt.
- **Mobile landscape orientation overlay** ŌĆö a full-screen `#landscape-overlay` element is shown on touch devices in landscape orientation with a viewport height under 500px (phones, not tablets). The overlay uses `var(--bg)` and `var(--text-muted)` to follow the active theme and displays an animated rotate icon prompting the user to return to portrait. It is purely CSS-driven via `@media (orientation: landscape) and (pointer: coarse) and (max-height: 500px)` and requires no JavaScript.
- **Mobile composer layout broken after app backgrounding** ŌĆö returning to the browser after switching to another app (or the home screen) left the mobile composer and keyboard helper bar pinned near the top of the screen with a large gray gap. The OS closes the keyboard when the app is backgrounded, but on iOS the `visualViewport` `resize` event does not always fire on return, so the stale `mobile-keyboard-open` class and `--mobile-keyboard-offset` value persisted. A `visibilitychange` listener added to `bindMobileComposerKeyboardListeners` re-runs `syncMobileComposerKeyboard()` followed by `syncMobileViewportState()` after a 50 ms settle delay whenever the page transitions from hidden to visible. `syncMobileViewportState` calls `isMobileKeyboardOpen()` which returns `false` when the offset is at or below the baseline, clearing the stale class and restoring the correct layout.

### Changed
- **Composer state Phase 3** ŌĆö focus and selection changes now publish directly into `composerState`, and the last hidden-input mirroring helper was removed. Mobile focus, desktop focus, and selectionchange events now update shared composer state instead of pushing DOM state from one input into the other.
- **Mobile composer-host spacing guard** ŌĆö added a unit regression that reads `styles.css` and asserts the simplified mobile shell does not put keyboard-height spacing back onto `#mobile-composer-host`. This keeps the stale gap fix from being silently reintroduced during later mobile-shell tuning.
- **Composer state Phase 2** ŌĆö `setComposerValue()` now renders only the active input and dispatches only to that target, while the mobile history/autocomplete paths publish through the shared `composerState` instead of mirroring into the hidden desktop input. The selection-change and mobile tap helpers were tightened to update shared state directly, which keeps desktop and mobile composer ownership one-way without reintroducing the mobile shell coupling that the Firefox fix removed.
- **iptables loopback block for scanner user** ŌĆö `entrypoint.sh` now adds an iptables `OUTPUT` rule (before the `gosu` drop) that `REJECT`s TCP connections from the `scanner` uid to the app port with a TCP reset. This provides an OS-level backstop for tools that could reach internal endpoints even if they bypass command validation (e.g. scripting languages, `nc`). `iptables` is now installed in the Dockerfile. The rule uses `|| true` so startup is not aborted in environments where `xt_owner` is unavailable.
- **Configurable app port (`APP_PORT`)** ŌĆö the application port is now controlled by a single `APP_PORT` environment variable (default `8888`). `Dockerfile` uses `ARG APP_PORT` / `EXPOSE ${APP_PORT}`; `entrypoint.sh` uses `${APP_PORT:-8888}` for both the gunicorn bind address and the iptables rule; both `docker-compose.yml` and `examples/docker-compose.standalone.yml` use `${APP_PORT:-8888}` for `expose`/`ports`, build args, the `APP_PORT` environment variable passed into the container, and the healthcheck URL. A `.env` file is included in the repo with the default (`APP_PORT=8888`); edit it to move the whole stack to a different port. The inline `:-8888` fallbacks in `docker-compose.yml` and `ARG APP_PORT=8888` in `Dockerfile` are safety nets only for bare `docker build` runs that bypass Compose. When running `docker compose -f examples/docker-compose.standalone.yml` from the project root, pass `--env-file .env` explicitly because Compose looks for `.env` in the same directory as the compose file.
- **Playwright e2e test commands** ŌĆö all Playwright specs that previously used `curl http://localhost:5001/...` as the test command were updated to use built-in fake shell commands (`hostname`, `date`, `uptime`, `whoami`, etc.). The `curl localhost` pattern triggers the `_LOOPBACK_RE` block added in this release, causing those tests to fail immediately. Fake commands bypass both the allowlist check and the loopback block because `/run` resolves them before calling `is_command_allowed()`. Tests that run two consecutive commands back-to-back (`clear all history` in `history.spec.js`, `Delete Non-Favorites` in `history.spec.js`, `clearing history` in `failure-paths.spec.js`) also received a `page.waitForTimeout(1200)` between commands: fake commands complete in ~5ms, so without the gap both runs land in the same 1-second rate-limit window under the default `rate_limit_per_second: 5` config. The `history.spec.js` `beforeEach` also gained a `page.reload()` after `localStorage.clear()` to ensure a clean state on each run.
- **`diagnostics_allowed_cidrs` guidance updated** ŌĆö the `config.yaml` comment for this setting now recommends against including `127.0.0.1/32` and explains that any process inside the container (including web shell commands running as `scanner`) connects from `127.0.0.1`. Host browser and curl access arrives as the Docker bridge gateway IP instead, so `172.16.0.0/12` is the correct scope for operator access.
- **Backend route decomposition** ŌĆö `app/app.py` has been split into focused Flask Blueprints under `app/blueprints/`: `assets.py` (vendor assets, favicon, health check), `content.py` (index, config, themes, FAQ, autocomplete, welcome endpoints), `run.py` (command execution and kill, with all run-output helpers and the rate-limited `/run` route), and `history.py` (history and share permalink routes with preview-output helpers). Shared per-request helpers live in `app/helpers.py` (trusted-proxy IP resolution, session ID extraction) and the Flask-Limiter singleton in `app/extensions.py`. `app/app.py` is now a thin factory that configures logging, calls `limiter.init_app`, registers blueprints, and wires the before/after-request logging hooks and the 429 error handler. Route behavior, response formats, status codes, rate limits, and SSE streaming are unchanged.
- **Frontend controller split** ŌĆö `app/static/js/app.js` now keeps the shared UI helpers, while the page bootstrap, config loading, modal and search/history wiring, and mobile composer setup live in `app/static/js/controller.js`. The browser template and jsdom harness both load the controller after `app.js`, preserving the existing classic-script globals while separating the composition root from the reusable helpers.
- **Global-state and script-order coupling cleanup** ŌĆö `app/static/js/state.js` now owns the shared store boundary and explicit tab state accessors, while DOM-facing helpers live in `app/static/js/ui_helpers.js`. The browser template and jsdom harness load that helper layer explicitly before the DOM-bound modules that depend on it, which keeps the classic-script architecture intact while reducing implicit ordering requirements.
- **DOM construction and template cleanup** ŌĆö the repeated tab-history and FAQ-limit rendering paths now use direct DOM construction instead of stitched HTML strings, and the shell modal chrome moved to class-based hidden/dialog wrappers in the template. The visible UI and accessibility behavior are unchanged, but the markup is easier to maintain and less brittle.
- **Search highlighting hardening** ŌĆö `app/static/js/search.js` now highlights matches by walking text nodes and cloning the line structure instead of rewriting serialized `innerHTML`, so mixed-content lines keep their helper markup intact while plain-text search, regex mode, case sensitivity, and current-match navigation behave as before.
- **FAQ inline style cleanup** ŌĆö the three inline `style="color:var(--...)"` attributes and the `<ul style="...">` layout attribute in the built-in FAQ entries in `app/commands.py` were replaced with CSS classes (`.faq-link` for green documentation links, `.faq-kill-verb` for the red Kill label), removing presentation details from the content layer. The keyboard-shortcuts `<ul>` inline style was removed entirely since the existing `.faq-a ul` rule already handles the layout.
- **Fake command dispatch map** ŌĆö the 30-way `if/elif` chain in `execute_fake_command()` in `app/fake_commands.py` was replaced with a module-level `_FAKE_COMMAND_DISPATCH` dict of lambdas normalised to `(cmd, sid) -> list[dict]`. The public function body is now four lines: resolve the root, look up the handler, return the unsupported-command error if absent, or call the handler. Behavior is unchanged.
- **Autocomplete above-prompt item order** ŌĆö removed the extra reversal logic that rendered the suggestion list bottom-to-top and flipped `ArrowDown`/`ArrowUp` direction when the dropdown appeared above the prompt. Items now render top-to-bottom in their natural order regardless of dropdown position, matching the Ctrl+R hist-search dropdown and eliminating the inconsistent navigation direction when the list flipped above.

Format follows [Keep a Changelog](https://keepachangelog.com/en/1.0.0/).

## [1.4] — unreleased

### Fixed
- **Loopback address blocking** — commands containing `localhost`, `127.0.0.1`, `0.0.0.0`, or `[::1]` anywhere in the command string are now blocked by `_LOOPBACK_RE` in `commands.py`, using the same pattern as the existing `/data` and `/tmp` path checks. This prevents web shell users from curling internal Flask endpoints (e.g. `curl http://localhost:8888/diag` or `curl 127.0.0.1:8888/config`) regardless of how the URL is formatted. Word-boundary anchors prevent false positives on external hostnames like `notlocalhost.com`. Client-side deny entries (`!curl localhost`, `!curl 127.0.0.1`, `!wget localhost`, etc.) in `conf/allowed_commands.txt` provide immediate browser feedback for the most obvious bare-hostname patterns.
- **History drawer closes on typing** — typing in the command input while the run history drawer was open no longer left the drawer visible. The `input` event handler in `controller.js` now calls `hideHistoryPanel()` when the drawer is open, matching the behavior of submitting a command or opening a modal.
- **Modal overlay keyboard blocking** — the FAQ, Options, and Theme modals now block all keyboard input from reaching the terminal, not just printable characters. Previously, pressing `Enter` while a content modal was open appended a new prompt line to the terminal output, and `Ctrl+C` could trigger kill-confirmation on a running tab. Both the `cmdInput` keydown handler and the `document` keydown handler in `controller.js` now return early for all keys (except `Escape`) when any of those three overlays are open. `Escape` still closes the active overlay and refocuses the terminal input.
- **App name font consistency across pages** — the app name in the header of permalink and diag pages was using `var(--font-display)` (Syne) at 16px glow, while the main shell uses `var(--font-mono)` (JetBrains Mono) at 20px glow. The `.permalink-title` rule in `styles.css` now uses `var(--font-mono)`, 20px glow, and the same weight and letter-spacing as the main header so the app name renders identically across all pages.
- **Permalink and diag action button hover** — the `← shell` link and other action buttons on permalink and diag pages had no hover highlight. A scoped `.permalink-actions .term-action-btn:hover` rule was added to `styles.css` to apply the same green border and text color used by action buttons on the main shell.
- **Gunicorn orphan-process crash (`init: true`)** — added `init: true` to both `docker-compose.yml` and `examples/docker-compose.standalone.yml` so Docker's bundled tini process runs as PID 1. Without this, scanner tools like `wpscan` that exit with code 3 as a normal status code (meaning "potentially interesting findings") could crash the server: when the entire subprocess chain receives SIGTERM simultaneously, intermediate parents may die before the leaf tool, orphaning it to PID 1 (Gunicorn). Gunicorn's `reap_workers()` collects the orphan via `waitpid(-1)` and interprets `exit(3)` as `WORKER_BOOT_ERROR`, halting the server. With `init: true`, tini is PID 1, Gunicorn is PID 2+, and orphaned scanner processes are silently reaped by tini.
- **`/kill` endpoint PGID race condition** — the endpoint now uses the stored subprocess PID directly as the PGID instead of calling `os.getpgid(pid)` at kill time. Since all subprocesses are spawned with `preexec_fn=os.setsid`, PGID equals PID at creation, so the stored value is always correct. The previous `os.getpgid()` call would return the wrong PGID if the subprocess had already been reaped and its PID reused by a new Gunicorn worker, causing `kill -TERM -<worker_pgid>` to send SIGTERM to the entire Gunicorn worker pool.
- **Late kill / exit prompt race** — when a command finishes at the same time the user clicks Kill, the runner now re-attempts deferred prompt mounting after the kill path so the active tab does not get stuck without the prompt below the output. The same prompt-remount path also runs when the late exit event arrives after a kill click.
- **Container Smoke Test DinD filesystem mismatch** — the `container_smoke_test` fixture no longer depends on runtime bind mounts that assume the Docker daemon can see the test process's filesystem. It now builds the base image, creates a temporary container, copies the repo `app/` tree plus a generated `config.local.yaml` into `/app`, commits that as a runtime image, and starts compose from that image with no client-side bind mounts. This fixes the GitLab DinD case where the daemon runs in a separate sidecar and cannot reliably mount paths from the job container.
- **Container Smoke Test DinD port selection** — the fixture no longer guesses a free localhost port from the job container namespace. It now publishes `8888` on an ephemeral host port and discovers the real published port with `docker compose port`, which avoids cross-namespace port collisions under DinD.
- **SQLite bootstrap race under Gunicorn startup** — `db_init()` now takes a filesystem lock before schema creation, migration, and retention pruning. Without that lock, multiple Gunicorn workers importing `database.py` at the same time could race through `PRAGMA journal_mode=WAL` and startup migration work, intermittently failing with `sqlite3.OperationalError: database is locked`.
- **Search bar hidden-on-load** — `ui_helpers.js` now sets `searchBar.style.display = 'none'` at initialisation so the inline style overrides the conflicting `.search-bar { display: flex }` rule (same specificity, later in the stylesheet) that was causing the bar to appear visible on load when `.u-hidden` was also applied. `isSearchBarOpen()` was also tightened from `!== 'none'` to `=== 'flex'` so the bar is treated as closed before any JS interaction sets the inline style explicitly.
- **Blank Enter while a command is running** — pressing Enter on an empty prompt while a tab is in the `running` state no longer appended a spurious blue prompt line to the output. The empty-command branch in `submitCommand` now returns early when the active tab is running.
- **Deny-prefix flag matching rewrite** — `_is_denied()` now tokenizes both the command and deny entries using the shared `split_command_argv` helper, matches tool and subcommand prefixes case-insensitively, and matches flags with exact case. Previously, the entire deny entry was lowercased before comparison, so `!curl -K` also blocked `curl -k`. The new implementation also handles combined short flags (e.g. `-sU` is caught by a `!nmap -sU` deny entry) correctly.
- **Draft input corrupted when switching tabs with a keyboard shortcut** — pressing a keyboard shortcut that contains the Tab key (such as `Alt+Tab`) while `cmdInput` had focus triggered the autocomplete accept handler (`acAccept`) because the `Tab` key case in the `cmdInput` keydown handler had no modifier guard. If `acFiltered` still held stale suggestions from the previous tab's typing session (e.g. `['whoami']`), `acAccept` would overwrite the current input and fire an input event that saved the wrong value into the active tab's `draftInput`. On the next switch, that tab's draft was restored as the stale suggestion instead of the original command. Fixed by adding `!e.altKey && !e.ctrlKey && !e.metaKey` guards to the `Tab` handler in `controller.js` so modifier-key combinations no longer trigger autocomplete logic.
- **Stale autocomplete suggestions persisting across tab switches** — `activateTab` in `tabs.js` now calls `acHide()` and resets `acFiltered = []` when switching tabs, so suggestions filtered for one tab's typing session cannot bleed into another tab. Without this, pressing bare Tab on a newly activated tab could accept a suggestion from the previous tab's last input instead of filtering for the current draft.
- **Ctrl+R hist-search auto-fill** — typing in hist-search mode immediately overwrote the input with the first matched command, so continuing to type appended to the full match instead of refining the query (e.g. typing `w` matched `whoami` and populated the field, so typing `ho` then produced `whoamiho`). `handleHistSearchInput` no longer calls `setComposerValue` during typing; the typed query stays in the input while matches appear in the dropdown. ArrowDown/Up and Ctrl+R navigate the list and fill the input with the highlighted match; Tab accepts without running; Enter accepts and runs.
- **Ctrl+C in hist-search left autocomplete open** — pressing Ctrl+C to exit hist-search called `exitHistSearch(false)`, which restored the pre-search draft via `setComposerValue` with `dispatch: true`. That fired an input event that triggered `handleComposerInputChange`, reopening the autocomplete dropdown if `acFiltered` had entries. A second Ctrl+C was required to dismiss it. Fixed: `exitHistSearch` now accepts a `keepCurrent` option. The Ctrl+C handler passes `{ keepCurrent: true }` so the typed query stays in the input instead of restoring the pre-search draft. `exitHistSearch` also calls `acHide()` unconditionally at the end so autocomplete cannot reopen regardless of the exit path.
- **Hist-search ArrowDown/Up fell through to command history navigation** — pressing ArrowDown/Up while hist-search was active was not handled by `handleHistSearchKey`, so the keys fell through to the command history navigator instead of navigating the hist-search dropdown. Added ArrowDown/Up cases that step `_histSearchIndex` through the match list, fill the input with the selected match via `setComposerValue`, and re-render the dropdown highlight.
- **Hist-search Tab accepted from the wrong source** — Tab while hist-search was active fell through to the `acFiltered`-based autocomplete Tab handler in `controller.js`, which could accept a stale suggestion unrelated to the hist-search results. Added a Tab case to `handleHistSearchKey` that calls `exitHistSearch(true)` to accept the currently selected hist-search match and close the dropdown without running the command.
- **Hist-search Enter did not run the command** — `exitHistSearch(true)` on Enter filled the input with the selected match but never called `submitComposerCommand`, so the menu closed without executing anything. The Enter handler now calls `submitComposerCommand(cmdInput.value, { dismissKeyboard: true })` after accepting. When there are no matches it exits with `keepCurrent: true` (preserving the typed query) and runs that instead of restoring the pre-search draft.
- **Hist-search typing appended to pre-search draft** — `enterHistSearch()` did not clear the input when activating search mode, so if the user had a command like `ping -c 10 google.com` in the input and pressed Ctrl+R, the first keystroke would append to that text rather than starting a fresh query. Fixed: `enterHistSearch` now saves the current input as `_histSearchPreDraft` and then clears it (without dispatching) so subsequent typing goes directly into the search query. Escape and Ctrl+G still restore the original draft.

### Changed
- **iptables loopback block for scanner user** — `entrypoint.sh` now adds an iptables `OUTPUT` rule (before the `gosu` drop) that `REJECT`s TCP connections from the `scanner` uid to the app port with a TCP reset. This provides an OS-level backstop for tools that could reach internal endpoints even if they bypass command validation (e.g. scripting languages, `nc`). `iptables` is now installed in the Dockerfile. The rule uses `|| true` so startup is not aborted in environments where `xt_owner` is unavailable.
- **Configurable app port (`APP_PORT`)** — the application port is now controlled by a single `APP_PORT` environment variable (default `8888`). `Dockerfile` uses `ARG APP_PORT` / `EXPOSE ${APP_PORT}`; `entrypoint.sh` uses `${APP_PORT:-8888}` for both the gunicorn bind address and the iptables rule; both `docker-compose.yml` and `examples/docker-compose.standalone.yml` use `${APP_PORT:-8888}` for `expose`/`ports`, build args, the `APP_PORT` environment variable passed into the container, and the healthcheck URL. A `.env` file is included in the repo with the default (`APP_PORT=8888`); edit it to move the whole stack to a different port. The inline `:-8888` fallbacks in `docker-compose.yml` and `ARG APP_PORT=8888` in `Dockerfile` are safety nets only for bare `docker build` runs that bypass Compose. When running `docker compose -f examples/docker-compose.standalone.yml` from the project root, pass `--env-file .env` explicitly because Compose looks for `.env` in the same directory as the compose file.
- **Playwright e2e test commands** — all Playwright specs that previously used `curl http://localhost:5001/...` as the test command were updated to use built-in fake shell commands (`hostname`, `date`, `uptime`, `whoami`, etc.). The `curl localhost` pattern triggers the `_LOOPBACK_RE` block added in this release, causing those tests to fail immediately. Fake commands bypass both the allowlist check and the loopback block because `/run` resolves them before calling `is_command_allowed()`. Tests that run two consecutive commands back-to-back (`clear all history` in `history.spec.js`, `Delete Non-Favorites` in `history.spec.js`, `clearing history` in `failure-paths.spec.js`) also received a `page.waitForTimeout(1200)` between commands: fake commands complete in ~5ms, so without the gap both runs land in the same 1-second rate-limit window under the default `rate_limit_per_second: 5` config. The `history.spec.js` `beforeEach` also gained a `page.reload()` after `localStorage.clear()` to ensure a clean state on each run.
- **`diagnostics_allowed_cidrs` guidance updated** — the `config.yaml` comment for this setting now recommends against including `127.0.0.1/32` and explains that any process inside the container (including web shell commands running as `scanner`) connects from `127.0.0.1`. Host browser and curl access arrives as the Docker bridge gateway IP instead, so `172.16.0.0/12` is the correct scope for operator access.
- **Backend route decomposition** — `app/app.py` has been split into focused Flask Blueprints under `app/blueprints/`: `assets.py` (vendor assets, favicon, health check), `content.py` (index, config, themes, FAQ, autocomplete, welcome endpoints), `run.py` (command execution and kill, with all run-output helpers and the rate-limited `/run` route), and `history.py` (history and share permalink routes with preview-output helpers). Shared per-request helpers live in `app/helpers.py` (trusted-proxy IP resolution, session ID extraction) and the Flask-Limiter singleton in `app/extensions.py`. `app/app.py` is now a thin factory that configures logging, calls `limiter.init_app`, registers blueprints, and wires the before/after-request logging hooks and the 429 error handler. Route behavior, response formats, status codes, rate limits, and SSE streaming are unchanged.
- **Frontend controller split** — `app/static/js/app.js` now keeps the shared UI helpers, while the page bootstrap, config loading, modal and search/history wiring, and mobile composer setup live in `app/static/js/controller.js`. The browser template and jsdom harness both load the controller after `app.js`, preserving the existing classic-script globals while separating the composition root from the reusable helpers.
- **Global-state and script-order coupling cleanup** — `app/static/js/state.js` now owns the shared store boundary and explicit tab state accessors, while DOM-facing helpers live in `app/static/js/ui_helpers.js`. The browser template and jsdom harness load that helper layer explicitly before the DOM-bound modules that depend on it, which keeps the classic-script architecture intact while reducing implicit ordering requirements.
- **DOM construction and template cleanup** — the repeated tab-history and FAQ-limit rendering paths now use direct DOM construction instead of stitched HTML strings, and the shell modal chrome moved to class-based hidden/dialog wrappers in the template. The visible UI and accessibility behavior are unchanged, but the markup is easier to maintain and less brittle.
- **Search highlighting hardening** — `app/static/js/search.js` now highlights matches by walking text nodes and cloning the line structure instead of rewriting serialized `innerHTML`, so mixed-content lines keep their helper markup intact while plain-text search, regex mode, case sensitivity, and current-match navigation behave as before.
- **FAQ inline style cleanup** — the three inline `style="color:var(--...)"` attributes and the `<ul style="...">` layout attribute in the built-in FAQ entries in `app/commands.py` were replaced with CSS classes (`.faq-link` for green documentation links, `.faq-kill-verb` for the red Kill label), removing presentation details from the content layer. The keyboard-shortcuts `<ul>` inline style was removed entirely since the existing `.faq-a ul` rule already handles the layout.
- **Fake command dispatch map** — the 30-branch `if/elif` chain in `execute_fake_command()` in `app/fake_commands.py` was replaced with a module-level `_FAKE_COMMAND_DISPATCH` dict of lambdas normalised to `(cmd, sid) -> list[dict]`. The public function body is now four lines: resolve the root, look up the handler, return the unsupported-command error if absent, or call the handler. Behavior is unchanged.
- **Autocomplete above-prompt item order** — removed the extra reversal logic that rendered the suggestion list bottom-to-top and flipped `ArrowDown`/`ArrowUp` direction when the dropdown appeared above the prompt. Items now render top-to-bottom in their natural order regardless of dropdown position, matching the Ctrl+R hist-search dropdown and eliminating the inconsistent navigation direction when the list flipped above.
- **Autocomplete and hist-search wrap-around navigation** — `ArrowDown` at the last item wraps back to the first; `ArrowUp` at the first item (or with no selection) wraps to the last. Both the autocomplete dropdown and the Ctrl+R hist-search dropdown now support indefinite cycling in either direction.
- **CSS section comments** — `app/static/css/styles.css` now has 22 consistent `/* ── Section ── */` markers covering every major surface (CSS variables, base reset, header, shell input, autocomplete, terminal, tab panels, history row, header buttons, modals, theme overrides, tabs, search, permalink toast, permalink pages, run history panel, starred chips, hamburger menu, welcome typeout, tab rename, mobile composer, mobile layout). The theme-override block (body-scoped selectors) also has an explanatory comment describing why `body .foo` specificity is needed over the `:root` defaults.
- **CSS duplicate selector cleanup** — removed 10 redundant `body`-prefixed rules from `app/static/css/styles.css` whose base counterparts already used the same `--theme-*` variables: `body .tabs-bar` (background + scrollbar), all three `body .tabs-bar::-webkit-scrollbar-*` rules, both `body .tabs-scroll-btn` hover/base rules, `body .ac-dropdown`, `body .ac-dropdown.ac-up`, `body .ac-item`, and `body .line.notice`. The two `.ac-dropdown` base box-shadows were updated to `rgba(0,0,0,0.08)` (what the now-removed body override had been enforcing via specificity) before removal. The two split `body.mobile-terminal-mode #mobile-composer-host` declarations were merged into one, dropping the dead first `background` value that the later declaration immediately overwrote.
- **CSS hardcoded color tokenization** — six hardcoded color literals scattered across component rules in `app/static/css/styles.css` were replaced with new `--theme-*` variables: `#000` on green-background elements (caret char, run button) → `--theme-on-accent-text`; selection highlight text colors `#f7fff2` and `#eef7ee` → `--theme-selection-text` and `--theme-selection-line-text`; modal button text `#fff` and `#000` → `--theme-modal-danger-btn-text` and `--theme-modal-warning-btn-text`. All six default values are defined in `:root` so future theme files can override them without touching component rules.

### Added
- **Ctrl+R reverse-history search** — pressing `Ctrl+R` in the command prompt enters reverse-i-search mode. A dropdown appears above the prompt (mirroring the autocomplete dropdown position) listing up to 20 history entries with the query substring highlighted. Typing narrows results and auto-fills the top match; pressing `Ctrl+R` again cycles to the next match; `Enter` accepts; `Escape` or `Ctrl+G` cancels and restores the draft that was in the prompt before search started. Implemented in a new section of `history.js` (`enterHistSearch`, `exitHistSearch`, `handleHistSearchInput`, `handleHistSearchKey`), with `controller.js` routing keydown events through `handleHistSearchKey` first when in search mode and intercepting `Ctrl+R` to enter search. DOM element `#hist-search-dropdown` added to `index.html` and `dom.js`; CSS in `styles.css`.
- **Per-tab draft input persistence** — composer text that has not been submitted is now saved per tab when switching away and restored when switching back. Drafts are not saved for running tabs (the command was submitted). Implemented via a `draftInput` field on each tab object in `tabs.js`, with `activateTab` flushing the leaving tab's draft from `cmdInput.value` directly (bypassing `getComposerValue()` to avoid a mobile/desktop selector race) and restoring the arriving tab's draft via `setComposerValue(..., { dispatch: false })`. The `controller.js` input handler also keeps `draftInput` live on every keystroke so the flush at switch time is always consistent.
- **Snapshot metadata block on permalink pages** — every `/history/<run_id>` and `/share/<id>` page now shows a metadata strip in the header. Run permalinks display an exit code badge (green border for exit 0, red for non-zero), duration (`1m 30s`), line count with full/preview/truncated scope, and app version. Snapshot permalinks show line count and app version (no exit code or duration — snapshots are tab captures with no single exit state). Downloaded HTML exports include the same metadata in their header via inline styles so the self-contained file stays portable. Implemented via a new `_format_duration()` helper in `permalinks.py`, a `meta` keyword argument on `_permalink_context()` and `_permalink_page()`, metadata construction in both `get_run()` and `get_share()` in `blueprints/history.py`, and new CSS classes in `styles.css`.
- **`rate_limit_enabled` config switch** — rate limiting can now be toggled via `config.yaml` or `config.local.yaml`. Defaults to `true`; set to `false` in a local overlay to bypass throttling during the Container Smoke Test corpus capture without touching production config. The `RATELIMIT_ENABLED` Flask-Limiter flag is set from this value at startup.
- **`Option+Tab` / `Alt+Tab` tab cycling** — `Option+Tab` (macOS) / `Alt+Tab` (Linux/Windows) cycles to the next tab; adding `Shift` reverses direction. Behaves identically to `Option+←/→` and is listed in the `shortcuts` helper output and FAQ.
- **Optional local config overlays** — sibling `*.local.*` files are now supported across the checked-in config assets, so operators can keep private per-server overrides out of git while leaving the base config and asset files portable for other deployments
- **One-hour command timeout default** — the checked-in config and built-in runtime default now set `command_timeout_seconds` to `3600`, so long-running commands are bounded by default instead of starting out disabled
- **Configurable prompt prefix** — `prompt_prefix` is now a first-class config setting, so operators can customize the terminal prompt text and welcome samples independently of the app name
- **Theme externalization** — the theme palette and component chrome values are now documented in the named YAML files under `app/conf/themes/`, injected into the live pages as shared CSS/JS theme variables, and reused by the permalink/export helpers so exported HTML stays in sync with the selected theme variant. The root `app/conf/theme_dark.yaml.example` and `app/conf/theme_light.yaml.example` files are copyable templates only.
- **Runtime theme selector** — the completed dedicated theme selector modal now exposes a grouped preview grid fed by the theme registry, with `label:`, `group:`, and `sort:` metadata controlling the visible names and section layout, so the live shell can choose among the named YAML variants under `app/conf/themes/` without a page reload while keeping permalinks and exports aligned with the selected theme
- **Operator diagnostics endpoint (`/diag`)** — a new IP-gated `/diag` route in `blueprints/assets.py` returns a themed HTML summary (or JSON with `?format=json`) of app version and name, operational config values (rate limits, timeouts, output limits, retention, proxy CIDRs, log settings), SQLite run/snapshot counts, Redis connectivity, vendor-asset source (built-in vs fallback), per-tool availability derived from the allowlist, and usage statistics (run counts by period, exit-code outcome breakdown, top 10 commands by frequency, and top 5 longest individual runs). The route returns 404 unless the direct TCP peer IP falls within the `diagnostics_allowed_cidrs` list in `config.yaml`; the list defaults to empty so the endpoint is off by default. Because it uses `request.remote_addr` directly rather than the resolved `X-Forwarded-For` client IP, `X-Forwarded-For` spoofing cannot bypass the gate — only requests whose connection originates from an allowed address are served. Denied access is logged as `DIAG_DENIED` with the peer IP and configured CIDRs; allowed access is logged as `DIAG_VIEWED`. A new `ip_is_in_cidrs()` helper was added to `helpers.py`, reusing the existing `_trusted_proxy_networks` CIDR cache. The page layout places all four status cards (App, DB, Redis, Vendor Assets) in a single equal-width row so cards stay compact; the Activity and Top Commands sections each use a two-column split below the status row.
- **Diag link in the main UI** — when the `/config` response includes `diag_enabled: true` (set when the visiting IP is in `diagnostics_allowed_cidrs`), a `⊕ diag` button appears in the desktop header alongside history/options/theme/FAQ, and a matching entry appears in the mobile hamburger menu. Both are hidden by default (`u-hidden`) and revealed only after the config fetch confirms access. The desktop button is an `<a>` element styled to match the other header buttons, including full participation in the theme-override CSS block so it follows `--theme-toolbar-button-*` variables consistently.
- **Mobile theme selector layout** — the theme picker now opens as a full-screen chooser on mobile with a two-column preview layout on wider phones, so the preview cards stay readable and the grouped sections keep a uniform width instead of collapsing unevenly on small screens

## [1.3] — 2026-04-07

### Added
- **Terminal-native input surface** — the visible command-entry UI is now rendered inline inside terminal output while keeping a hidden real input for browser/mobile keyboard semantics
- **Shell-like editing and control behavior** — blank/whitespace `Enter` now emits a fresh prompt line, and `Ctrl+C` now opens kill confirmation during active runs or emits a new prompt line when idle
- **Keyboard shortcut layer** — app-safe shortcuts now cover tab lifecycle, active-tab actions, and shell-style line editing
  - `Alt/Option+T`, `Alt/Option+W`, `Alt/Option+ArrowLeft/Right`, and `Alt/Option+1...9` manage tab creation, closing, and navigation
  - `Alt/Option+P`, `Alt/Option+Shift+C`, and `Ctrl+L` trigger permalink, copy, and clear actions for the active tab
  - `Ctrl+U`, `Ctrl+K`, and `Alt/Option+B` / `Alt/Option+F` add readline-style prompt editing on top of the existing `Ctrl+W` support
- **Keyboard shortcut discovery** — new `shortcuts` web-shell helper prints the current shortcut set and is exposed through `help`, autocomplete, the README, and the FAQ modal
- **Shortcut helper cleanup** — the old `keys` helper alias and fallback-notes wording were removed so `shortcuts` is the only supported name in help, autocomplete, docs, and in-terminal output
- **Bundled local fonts** — JetBrains Mono and Syne are now served through local vendor routes backed by build-time font downloads, with repo fallbacks for local dev, so the shell and permalink pages no longer make external font requests on load
- **Bundled local ansi_up** — the browser-facing `ansi_up` asset is now copied from the checked-in repo build into the image at build time, with the repo copy still serving as the local/docker-compose fallback
- **Template-backed permalink pages** — the live `/history/<id>` and `/share/<id>` pages now share Jinja templates plus shared CSS/JS instead of carrying duplicated inline HTML/CSS in `permalinks.py`, which makes the permalink UI easier to maintain while keeping the downloadable HTML export path self-contained
- **Shared HTML export helper** — tab `save html` and permalink export now share browser-side HTML/CSS helpers instead of duplicating export markup in two places, and downloaded tab exports embed vendor fonts at export time so the saved file stays portable
- **Mobile keyboard helper row race** — the mobile helper row now stays visible when viewport resize lands before the focus event, preventing the intermittent gray gap/scroll shift on touch keyboards
- **Tab overflow controls** — left/right scroll buttons were added to the tab bar for overflowed tab lists
- **Tab drag reorder** — tabs can now be reordered directly in the strip using drag-and-drop, including mobile touch drag with visual lift/drop indicators
- **FAQ markup syntax** — custom `faq.yaml` entries can now use lightweight markup for bold, italics, underline, inline code, bullet lists, and clickable command chips that load into the prompt
- **Autocomplete placement logic** — the suggestion list now supports above/below prompt placement and aligns to command start in the inline prompt model
- **Display toggles for output prefixes** — line numbers can now be toggled independently from timestamps, with mobile-menu access and shared prefix alignment across output, prompt, and exit rows
- **User options modal** — a new options modal lets users set timestamp mode and line-number display from one place, with cookie-backed persistence across sessions while keeping the existing quick-toggle buttons in sync. Theme selection lives in its own dedicated modal
- **Permalink display toggles** — permalink pages now expose line-number toggles for all views and timestamp toggles anywhere saved line metadata exists, including fresh canonical run permalinks backed by structured output persistence
- **Mobile composer UX** — the visible mobile composer now keeps Run, Enter-to-submit, history chips, autocomplete, and the edit-helper row wired to the same visible input so the touch keyboard path stays in sync
- **Mobile welcome assets** — added `ascii_mobile.txt` and `/welcome/ascii-mobile` for the dedicated mobile banner path

### Changed
- **FAQ single source of truth** — built-in FAQ entries now come from the backend alongside custom `faq.yaml` entries, `/faq` now returns the merged canonical FAQ dataset, and the modal renders from that backend response instead of a hard-coded HTML copy
- **Mobile welcome flow** — touch-sized screens now use `ascii_mobile.txt` with the same staged status and rotating-hint timing as desktop, while skipping the sampled-command phase from `welcome.yaml`
- **Full-output permalink behavior** — tab permalink snapshots now fetch the full persisted run output when it exists, so the main-page permalink button can share the complete result instead of the preview-only tab state

## [1.0] — initial release

### Added
- **Real-time command execution** — bash commands streamed to the browser over SSE; output rendered with ANSI color support via `ansi_up`
- **Multi-tab interface** — open multiple independent tabs, each with its own status, output, and kill button
- **Command allow/deny rules** — `allowed_commands.txt` whitelist with group labels; deny-prefix rules block dangerous flags (e.g. `--output`, `-oN`); `/dev/null` output flag allowed explicitly
- **Process kill** — per-tab ■ Kill button sends SIGTERM to the entire process group; confirmation modal prevents accidental kills; PID tracked in SQLite to avoid multi-worker race conditions
- **Run history panel** — last 50 runs per session stored in SQLite; load any result into a new tab; delete individual runs or clear all; session-isolated so users only see their own history
- **Permalinks** — share button saves a full tab snapshot to SQLite and returns a shareable URL; single-run permalinks also available from the history panel; styled HTML view with copy / save .txt options
- **Output search** — in-terminal search with case-sensitive and regex toggle, prev/next navigation, match count
- **Config YAML** — operator-configurable settings: `app_name`, `motd`, `default_theme`, `rate_limit_per_minute/second`, `command_timeout_seconds`, `max_output_lines`, `max_tabs`, `history_panel_limit`, `permalink_retention_days`, `heartbeat_interval_seconds`
- **Rate limiting** — Flask-Limiter backed by Redis (multi-worker safe); falls back to in-process memory if Redis is unavailable; real client IP auto-detected from `X-Forwarded-For` when valid, otherwise direct connection IP
- **Redis process tracking** — active PIDs stored in Redis (or in-process dict) so any Gunicorn worker can kill a process started by a different worker
- **Security model** — two non-root users: `appuser` runs the Flask/Gunicorn process; `scanner` runs all user commands; filesystem mounted read-only except `/tmp`; `sudo kill` used for cross-user SIGTERM
- **Gunicorn WSGI** — production server with configurable timeout (3600 s); heartbeat SSE comments prevent nginx/browser idle disconnects
- **Custom FAQ** — operator-supplied `faq.yaml` entries append after the built-in FAQ; lightweight markup and clickable command chips load commands directly into the input bar
- **Custom autocomplete** — `auto_complete.txt` drives the command input dropdown
- **MOTD** — optional message-of-the-day rendered in the header area
- **Theme selector** — named theme variants selected from the options modal; preference persisted in localStorage; operator can set `default_theme` to a registry filename if they want a preferred startup variant
- **Mobile menu** — hamburger menu exposes search, history, timestamps, theme, and FAQ on small screens
- **Docker support** — multi-stage Dockerfile; `docker-compose.yml` with Redis sidecar; health checks via `/health` endpoint; read-only root filesystem with tmpfs mounts
- **GitLab CI pipeline** — lint (flake8) and test (pytest) stages run on every push
- **Security tools** — nmap, masscan, naabu, httpx, nuclei, subfinder, dnsx, katana, nikto, wapiti3, wpscan, mtr (report mode), dig, host, whois, curl, ffuf, gobuster, feroxbuster and more
- **SecLists wordlists** — full collection installed at `/usr/share/wordlists/seclists/`
- **ARCHITECTURE.md** — documents the system design, data flow, module structure, and database schema
- **Unit test suite** — pytest tests covering command validation, route behaviour, rate limiting, and deny-rule logic; lint enforced via flake8
- **ARIA accessibility** — labelled inputs and buttons throughout the UI

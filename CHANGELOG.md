## [1.5] — Unreleased

### Added

- **`Alt+/` FAQ shortcut plus tooltip audit across every chrome button** — the FAQ overlay was the last chrome surface without a keyboard path; a new `Alt+/` chord opens and closes it via `handleChromeShortcut()` in `app.js` (matches `e.code === 'Slash'`, `e.key === '/'`, and the macOS Option glyph `÷`). Added as the seventh entry in the `UI` section of `_CURRENT_SHORTCUTS` (`app/fake_commands.py`), so it appears in both the `shortcuts` built-in command and the `?` overlay automatically. In the same pass, every button in the desktop shell chrome was audited for tooltip coverage: rail nav items (`history`, `options`, `theme`, `FAQ`) now list the matching Alt chord; the dynamically-built HUD action buttons in `shell_chrome.js` (`Kill`, `share snapshot`, `copy`, `save`, `clear`) gained `title` attrs (previously had none) with shortcut hints where one exists (`Option+P / Alt+P`, `Option+Shift+C`, `Ctrl+L`); `new-tab-btn` now advertises `(Option+T / Alt+T)`; the per-tab `✕` close button (built in `tabs.js`) now advertises `(Option+W / Alt+W when active)`. A stale `(Alt+F)` on `search-toggle-btn` was corrected to `(Alt+S)` to match the current binding. Mobile buttons that previously had only `aria-label` (`hamburger-btn`, `mobile-run-btn`, `mobile-kill-btn`) gained matching `title` attributes so tooltips work on tablets/styluses. Rail section headers (Recent, Workflows) gained titles for their collapse/expand behaviour. One new Playwright case: `Alt+/ toggles the FAQ overlay from the composer` in `tests/js/e2e/shortcuts.spec.js`.
- **Desktop chrome keyboard shortcuts** — six new Alt chords now toggle the rail and the four major overlay surfaces without touching the mouse: `Alt+\` toggles the rail collapsed/expanded, `Alt+S` toggles the transcript search bar, `Alt+H` toggles the history drawer, `Alt+,` toggles the options panel, `Alt+Shift+T` toggles the theme selector, and `Alt+G` toggles the guided workflows panel. Search is bound to `Alt+S` rather than `Alt+F` because the composer owns `Alt+F` as readline word-forward; `Alt+S` has no readline conflict and works identically from the prompt and from anywhere else in the UI. Each chord is a true toggle — pressing it while the surface is open closes the surface, and opening any of these surfaces while another is open swaps cleanly via the existing `_closeMajorOverlays()` path. Bindings are matched through a new `handleChromeShortcut()` dispatcher in `app.js` that accepts both `e.code` (for punctuation keys like `Backslash` and `Comma`) and the macOS Option-letter glyphs (`˙` for Alt+H, `ƒ` for Alt+F, `«` for Alt+\, etc.), so any keyboard layout lands on the same result. The dispatcher is called from the document keydown chain in `controller.js` after `handleTabShortcut` and `handleActionShortcut`, and it is also allowed through the major-overlay early-return gate so the same chord that opened a surface can close it again. `handleTabShortcut` was tightened with `!e.shiftKey` guards on its Alt+T and Alt+W branches so Alt+Shift+T falls through to the chrome handler instead of being eaten by the tab dispatcher. `shell_chrome.js` now exports `toggleRailCollapsed` from its IIFE so Alt+\ can drive it. For the toggle-close paths, the chrome handler calls `hideHistoryPanel()` / `closeOptions()` / `closeThemeSelector()` / `closeWorkflows()` directly rather than re-using the header-button `.click()` paths — those handlers call `_closeMajorOverlays()` first and would close-then-reopen the same surface. Button tooltips in the header and rail were updated in `templates/index.html` to advertise each binding (`Run history (Alt+H)`, `Theme selector (Alt+Shift+T)`, `Collapse sidebar (Alt+\)`, etc.), and the six bindings were added to `_CURRENT_SHORTCUTS` in `fake_commands.py` so they automatically appear in both the `shortcuts` built-in command output and the `?` keyboard-shortcuts overlay. Playwright coverage: seven new cases in `tests/js/e2e/shortcuts.spec.js` under `test.describe('desktop chrome keyboard shortcuts')` covering each chord's toggle behavior plus a negative test confirming chrome shortcuts are ignored while the composer is focused.
- **Dedicated keyboard-shortcuts overlay with a single source of truth** — pressing `?` from anywhere on the page now opens a transparent, keyboard-first overlay listing every shortcut. The binding works from the command prompt itself when the prompt is empty; once any text is in the prompt (or any other input), `?` types normally so args like `curl "…?foo=bar"` are not interfered with. The `shortcuts` built-in command and the overlay both read from a new `GET /shortcuts` JSON endpoint backed by `get_current_shortcuts()` in `fake_commands.py`, so the reference cannot drift between the terminal output and the UI surface. The overlay lists its own `?` trigger as the first entry, so the binding is self-documenting. The `?` handler skips modifier chords (Ctrl/Meta/Alt) and the welcome-animation active state — pressing `?` during welcome continues to settle the intro instead of opening the overlay. Escape and a backdrop click close the overlay. The FAQ's previously-long shortcut list has been reduced to a short pointer (`press ?` / `run shortcuts`) now that both surfaces share one source. Playwright coverage: five new cases under `test.describe('? keyboard-shortcuts overlay')` in `tests/js/e2e/shortcuts.spec.js` — `? opens the overlay when no input is focused`, `Escape closes the overlay`, `? opens the overlay from the empty command prompt`, `? types normally when the command prompt already has text`, and `overlay and shortcuts built-in share the same source`. Pytest coverage: four new cases under `TestShortcutsRoute` in `tests/py/test_routes.py` including a `test_matches_shortcuts_builtin_source` assertion that pins the endpoint payload to the `shortcuts` built-in source.
- **UI screenshot capture pipeline for design review, theming, and visual QA** — new standalone Playwright projects (`tests/js/e2e/ui-capture.desktop.capture.js`, `tests/js/e2e/ui-capture.mobile.capture.js`, plus a shared `ui_capture_shared.js` scene registry) drive a curated walk through both the desktop and mobile shell and emit a labelled PNG pack plus a per-UI manifest JSON. Run via `scripts/capture_ui_screenshots.sh` (also exposed as `npm run capture:ui-screenshots`) with optional `--ui desktop|mobile`, `--theme <slug>|all`, and output-directory overrides; defaults write to `test-results/ui-capture/`. The captures are gated by `RUN_CAPTURE=1` and live under dedicated configs (`config/playwright.capture.desktop.config.js`, `config/playwright.capture.mobile.config.js`) so they never run in the normal `test:e2e` suite. A new `scripts/seed_history.py` populates history.db with ~70 realistic runs under a UUID or `tok_` session so HUD status, `/diag`, recents, and history-heavy states render the way they would in a production instance; the capture harness uses it to pre-seed a well-known session token (`tok_cafebabecafebabecafebabecafebabe`) into each capture slot's ephemeral data directory. A new `APP_FAKE_REDIS=1` env var in `app/process.py` installs an in-memory Redis subset (`_FakeRedisClient` with `ping`/`set`/`get`/`getdel`/`sadd`/`smembers`/`expire`/`srem`/`delete` plus per-key TTL purging) so capture runs can exercise Redis-dependent code paths without booting a real Redis instance — `scripts/playwright/run_e2e_server.sh` now detects the `capture-*` slot prefix and opts into both the seeder and the fake client. The seeder is also documented in `tests/README.md` as a manual-QA helper for exercising the history drawer, reverse-i-search, filters, and token-migration workflows; it refuses to write from the host by default because the container SQLite and host SQLite can disagree on FTS5 internals (see the `sqlite_cross_version_fts` memory rule).

- **Reusable ambient outside-click helper replaces 6 hand-rolled document-level dismissal handlers and encodes the trigger-exemption contract (Phase 4b of the UI Interaction Helper Refactor)** — a new `bindOutsideClickClose(panel, { triggers, isOpen, onClose, exemptSelectors, scope })` helper in `app/static/js/ui_outside_click.js` is now the single contract for ambient outside-click dismissal of dropdowns and menus. Companion to Phase 4a's `bindDismissible`: `bindDismissible` owns scrim/modal-click dismissal for backed surfaces; `bindOutsideClickClose` owns document-level (or scope-overridden) click dismissal for unbacked panels where the trigger sits outside the dismissible surface. Same shape as prior helpers: one function, element + options bag, null-guards its preconditions, returns a `{ dispose }` handle. Behavior contract: `panel` may be `null` for sibling-set cases (multiple peer dropdowns sharing a parent — e.g. the two recents filter dropdowns on mobile), in which case containment is decided purely by `triggers` / `exemptSelectors`; `triggers` accepts a single element or array and is matched via `.contains()` (so clicks anywhere inside a trigger do not close); `exemptSelectors` is matched via `.closest()` against the click target (so clicks landing on or inside any matching ancestor do not close — used for `.hist-chip-overflow` + `[data-action="history"]` on the history panel, `.ac-dropdown` + `#mobile-composer` for autocomplete, `[data-recents-dropdown]` for the recents filter dropdowns); `scope` defaults to `document` but accepts any element (the recents sheet passes itself so only sheet-scoped clicks are evaluated, keeping outer-document clicks from reaching the handler); the handler short-circuits when `isOpen()` returns false so a no-op close is never triggered. Trigger exemption explicitly replaces the hand-rolled `e.stopPropagation()` pattern that every pre-refactor outside-click site used — trigger clicks now flow naturally, and the helper recognises them as part of the surface rather than as "outside" clicks. Migrated in this phase (six sites): the three near-identical save-menu dropdowns (`tabs.js` per-tab save-menu inside `_createTabPanel`, `shell_chrome.js` HUD save-menu, `permalink.js` save-menu) each collapse a hand-rolled document click listener plus a hand-rolled `e.stopPropagation()` on the trigger into one `bindOutsideClickClose` call with `triggers: saveBtn`; `controller.js` global-dismiss block (three stacked document click listeners for autocomplete, history panel, and mobile menu) collapses to two `bindOutsideClickClose` calls — history panel with `triggers: histBtn` + `exemptSelectors: ['.hist-chip-overflow', '[data-action="history"]']`, autocomplete anchored on `shellPromptWrap` with `exemptSelectors: ['.ac-dropdown', '#mobile-composer']` — with the mobile-menu outside-click deliberately removed as redundant now that the Phase 4a scrim (via `bindDismissible`) covers the viewport and handles dismissal there; `mobile_chrome.js` recents filter dropdowns (two sibling dropdowns on one sheet) collapse to a single `bindOutsideClickClose(null, { scope: recentsSheet, exemptSelectors: ['[data-recents-dropdown]'], ... })` with panel `null` because there is no single containing element — the sibling set is defined entirely by exemption. Old hand-rolled pieces removed: the three document click handlers in `tabs.js` / `shell_chrome.js` / `permalink.js`; the three `e.stopPropagation()` calls on save-menu triggers; the three-block Global click block in `controller.js`; the `recentsSheet.addEventListener('click', ...)` handler and the two `e.stopPropagation()` calls on recents dropdown triggers and option clicks in `mobile_chrome.js`. Post-action close semantics: the old document-level listener in `tabs.js` unconditionally removed `open` on any click (including clicks inside the menu on save-txt/save-html/save-pdf option buttons), so the menu auto-closed after each action; the outside-click helper intentionally does not fire on inside clicks, so `permalink.js` and `tabs.js` now explicitly call `wrap.classList.remove('open')` inside their data-action dispatchers after a save action — matching the explicit `saveWrap.classList.remove('open')` that `shell_chrome.js` already did per-option. Test coverage: new `tests/js/unit/ui_outside_click.test.js` with 19 cases covering guard paths (`panel: null` accepted, missing `opts` / `isOpen` / `onClose` each return `null`, handle exposes `dispose`), outside-click dismissal (closes outside, skips inside, skips on element itself, respects `isOpen`), trigger exemption (single, array, nested via `.contains()`, falsy entries ignored), exempt selectors (single, nested via `.closest()`, array), scope override, and handle `dispose()` on document and scope-override paths. Test harness (`tests/js/unit/helpers/extract.js`) extended with `UI_OUTSIDE_CLICK_SRC` bundled after `UI_DISMISSIBLE_SRC` in both `fromDomScript` and `fromDomScripts` so downstream module tests see `bindOutsideClickClose` as a global. Both templates (`index.html` and `permalink_base.html`) load `ui_outside_click.js` after `ui_dismissible.js`. Net suite delta: 566 Vitest (+19), 836 pytest, 171 Playwright all pass. Phases 5–6 (direct focus API + test and documentation follow-through) remain scoped in `TODO.md`.
- **Reusable dismissible helper replaces 11 hand-rolled modal/sheet dismissal paths and unifies the Escape cascade (Phase 4a of the UI Interaction Helper Refactor)** — a new `bindDismissible(el, { level, onClose, closeButtons, closeOnBackdrop, backdropEl, stopPropagation })` helper in `app/static/js/ui_dismissible.js` is now the single contract for scrim-backed overlay / modal / sheet surfaces. Behavior contract: `level` is one of `'modal' | 'sheet' | 'panel'` with priority `modal > sheet > panel`; within a level, the most recently registered open surface wins; `isOpen()` defaults to `!el.hidden && !el.classList.contains('u-hidden')` but callers can override for surfaces with bespoke visibility (e.g. history drawer's `.open` class); `closeButtons` accepts a single element or an array; `closeOnBackdrop` defaults to `true` with the default backdrop being the surface itself (click lands on `e.target === el`) — sheets with a detached scrim pass `backdropEl: <scrim>` so clicks on the scrim dismiss but clicks inside the sheet body don't; passing `closeOnBackdrop: false` or `backdropEl: null` disables backdrop-click entirely (used by the history drawer because it's a side panel, not a modal). Migrated in this phase: nine overlays registered through a new `setupDismissibleOverlays()` in `controller.js` (workflows, faq, theme, options, shortcuts as `level: 'modal'`; historyPanel as `level: 'panel'` with `closeOnBackdrop: false`; killOverlay, histDelOverlay, shareRedactionOverlay as `level: 'modal'`) plus two mobile bottom sheets in `mobile_chrome.js` (menu sheet + recents sheet, both `level: 'sheet'`, both with their scrim passed as `backdropEl`). Net suite delta: 547 Vitest (+29), 836 pytest, 171 Playwright all pass. Phase 4b (outside-click dismissible helper for ambient-click menus) and Phases 5–6 remain scoped in `TODO.md`.

- **Mobile output text overflowed off-screen when timestamps or line numbers were enabled** — the `.line-content` shift that makes room for the timestamp/line-number prefix was implemented with `transform: translateX(calc(var(--output-prefix-width) + 1ch))`. `transform` shifts where an element is *painted* but does not change its layout width, so `.line-content` retained the full `.output` content-box width and long lines (common with `nmap`, `curl`, `dig`) extended past the right edge of mobile viewports instead of wrapping at the shifted-in-from-the-left content edge — the exact bug described in the TODO under known issues. Padding reduces the content-box width so `white-space: pre-wrap` wraps text at the correct edge. The absolutely-positioned `::before` prefix (width `var(--output-prefix-width)`, padding-right `1ch`) anchors to the padding-box left edge of the positioned parent and renders inside the padding area, so the visual placement of the prefix is unchanged. If migration fails, the old session stays active and an error is shown; the newly generated token is discarded. - **`_seedLocalStorageStarsToServer` no longer silently drops stars on network failure** — failed `POST /session/starred` requests were swallowed and `localStorage.starred` was unconditionally deleted afterward, creating a data-loss window on flaky networks during first token activation. The function now uses `Promise.allSettled`, checks each response's `ok` flag, and writes only the failed commands back to `localStorage`. The key is removed only when every POST succeeds.
- **`_waitForMigrateChoice` modal no longer accumulates stale button listeners** — each call to `_waitForMigrateChoice` added new `click` handlers to the yes/no buttons without removing previous ones. If the modal was opened multiple times without a click (e.g. rapid re-triggers), stale `Promise` resolvers could fire on a later open. Fixed by replacing each button with a `cloneNode` copy before adding listeners, so no handler from a prior open survives.
- **`migrated_stars` response count now reflects newly inserted rows, not deleted rows** — `/session/migrate` used the `DELETE FROM starred_commands` rowcount for `migrated_stars`. When the destination session already had some of the same starred commands, `INSERT OR IGNORE` skipped them but `DELETE` still removed all source rows, causing the returned count to overstate how many commands were actually migrated. The count now uses the `INSERT OR IGNORE` rowcount, which only counts rows actually written to the destination.
- **Toast notifications now appear above the options modal** — the toast `z-index` was 300, below the options overlay at 320, so toasts triggered by session token operations (e.g. "token copied to clipboard") were hidden behind the open modal. Bumped to 400 so toasts render on top of all overlay surfaces.
- **Recent command chips and starred commands now reload immediately after any session token operation** — `cmdHistory` (the chip bar data) was populated only at page load and not refreshed when the active session changed. A new `reloadSessionHistory()` function in `history.js` sequences star-cache reload → history fetch → chip-bar hydration → history panel refresh (if open), and is called at every session switch path in `runner.js` and `controller.js`. Previously the chip bar and starred set only reflected the new session after a full page reload.
- **`session-token set` autocomplete no longer inserts the `<token>` placeholder** — the positional hint builder in `autocomplete.js` was not passing `insertValue` from YAML items to `_buildAutocompleteItem`, so the display label (`set <token>`) was also used as the inserted text. Both call sites now propagate `label` and `insertValue` from the YAML item; `session-token set` has `insertValue: "set "` in `autocomplete.yaml`, so `Tab` inserts `session-token set ` and leaves the cursor ready for pasting without needing to remove a placeholder.
- **Autocomplete placeholder hints (`<token>`, `<target>`, etc.) are now display-only across all commands** — three distinct issues combined to let `Tab` insert the literal placeholder text. First, the Python normalizer `_normalize_context_suggestion` in `app/commands.py` only preserved `value` and `description` from YAML items, silently dropping `insertValue`/`label`/`hintOnly` — so an entry like `{ value: "set <token>", insertValue: "set " }` reached the browser as `{ value: "set <token>" }` and the `set <token>` string itself was used as the insertion text (an earlier "fix" to propagate `insertValue` through `autocomplete.js` was a no-op because the key never arrived from the backend). Second, `_acItemInsertValue()` in `autocomplete.js` called `.trim()` on the insertion string, so even after the backend fix an author-intended trailing space in `insertValue: "set "` was stripped. Third, typing `session-token set ` and pressing `Tab` would insert the literal string `<token>` from the `arg_hints.set` placeholder — the builder could not distinguish a display-only placeholder from a real completion value. Fixes: `_normalize_context_suggestion` now preserves `insertValue` (whitespace-significant, untrimmed), `label`, and `hintOnly` when present; `_buildAutocompleteItem` auto-detects `<placeholder>` values (regex `^<[^<>\s][^<>]*>$`) as `hintOnly: true` when no explicit `insertValue` is given; `acAccept` short-circuits on `hintOnly` items (keyboard focus preserved; nothing inserted); a new `_acItemInsertText` helper used only by `acAccept` preserves intentional trailing whitespace in `insertValue`. Author opt-out: set `insertValue` explicitly on a `<placeholder>` entry to make it Tab-insertable, or set `hintOnly: false` to force. One new pytest case (`test_arg_hints_preserve_insert_value_with_trailing_whitespace`) covers the backend preservation; one new Vitest case covers the frontend (preserved whitespace, hintOnly detection, acAccept no-op on hintOnly).
- **Mobile tab bar now scrolls to keep the active tab fully visible** — `shouldScrollActiveTabIntoView()` was explicitly suppressing the `ensureActiveTabVisible()` call on mobile, meaning the active tab could be partially off-screen with no way to see it without manually scrolling. The guard was removed; the call is now unconditional across both desktop and mobile. `setTabLabel()` also now calls `ensureActiveTabVisible()` for the active tab so automatic renames triggered by command submission scroll the tab into view just as manual renames do.
- **Mobile autocomplete dropdown truncated and not scrollable with 3+ items** — two separate bugs combined to make the third (and any further) autocomplete item inaccessible on mobile. First, `touchstart` on each dropdown item called `e.preventDefault()`, which blocked the browser's native scroll gesture on the dropdown container — so touching the list to scroll it instead triggered the topmost item. Second, `maxHeight` in mobile mode was additionally capped by `available = rect.top - 12` (the vertical space above the composer), which could further clip the dropdown below the height needed to show all items. Fixed by replacing the `touchstart` + immediate-accept pattern with tap detection: `touchstart` now records the finger position passively (no `preventDefault`), and `touchend` only accepts the item if the finger moved less than 10px, letting swipe gestures fall through to native scroll. The secondary `available`-space cap was also removed so the dropdown expands to fit up to 8 items (360px max) without depending on a runtime geometry check.
- **`save txt` and `save html` no longer apply share redaction** — local file exports (the "save txt" and "save html" tab actions) were incorrectly running the same redaction pass as permalink sharing, replacing IPs, hostnames, emails, and bearer tokens in downloaded files. Redaction is now scoped exclusively to the permalink share flow where it belongs. `saveTab` now uses the raw exportable lines and `exportTabHtml` uses `rawLines` directly; only `permalinkTab` still calls `_getRedactedLines` before posting to `/share`.
- **Container smoke test now reports all failures instead of stopping after the first** — `scripts/container_smoke_test.sh` was passing `-x` to pytest, causing the run to abort on the first failure and leave remaining commands untested. The flag has been removed so the full suite runs and all failures are surfaced together.
- **Reverse-history search now searches full session history** — `Ctrl+R` previously searched only the small in-memory recent-commands chip list (capped at `recent_commands_limit`, default 8). It now fetches from the same `/history` endpoint the history drawer uses, applying the search query server-side before the `history_panel_limit` cap so commands from earlier in the session (or from previous days) are always reachable. Results are debounced per keystroke and capped at 10 in the dropdown. The dropdown no longer pre-populates on open — it only appears once the first character is typed.
- **Autocomplete examples not shown after accepting a command root** — accepting a suggestion via `Tab` or `Enter` (e.g. choosing `dnsenum` while typing `dns`) put the correct value in the prompt but did not show the command's example invocations. The root cause was that `acAccept` called `setComposerValue` before `acHide`: the input event fired synchronously, rendered the examples, and then `acHide` immediately closed them. Swapping the call order (hide first, then set value) lets the input event open the fresh autocomplete state without anything closing it afterward.
- **Example command domains corrected** — DNS-oriented tool examples in `autocomplete.yaml` and `scripts/smoke_test_commands.txt` were using `ip.darklab.sh` where they should use `darklab.sh` (a plain domain). Affected tools: `dig`, `nslookup`, `host`, `dnsrecon`, `subfinder`, `dnsx`, `dnsenum`, and `fierce`. Conversely, `nmap` and `curl` examples in `autocomplete.yaml` were using `darklab.sh` where they should use `ip.darklab.sh`. Both corrections are now applied consistently across both files.
- **Diagnostics redaction config rendering** — the `/diag` config table now renders numeric zero values as `0` instead of incorrectly treating them as boolean `false`, and the redaction-rule count field is now named `custom_redaction_rule_count` so it is clear the count only refers to operator-defined regex rules, not the built-in baseline.
- **`vitest.config.js` root path resolution** — after moving the config to `config/vitest.config.js`, the `root: '..'` option was being resolved relative to the working directory rather than the config file, causing vitest to look one level above the repo root. Fixed by computing the root path from `import.meta.url` so it is always the absolute repo root regardless of invocation directory.
- **naabu forced connect-scan mode** — `naabu` commands that do not already specify `-scan-type` or `-st` now have `-scan-type c` injected automatically before reaching the container. Raw SYN scanning requires libpcap and elevated capabilities that are not reliably available inside the container; TCP connect mode (`-sT` equivalent) produces the same open-port results without them.
- **Autocomplete stubs not discoverable via prefix matching** — empty specs in the autocomplete YAML (e.g. `whois: {}`) were silently dropped by `_normalize_autocomplete_context` because the guard condition only added a root to the registry when it had flags, arg_hints, or pipe metadata. Typing a partial name such as `whoi` returned an empty dropdown instead of suggesting `whois`. The guard was removed so all declared roots appear in `acContextRegistry` regardless of whether they carry additional spec data.
- **Deny-prefix combined short-flag matching** — `_is_denied()` now correctly matches combined POSIX short-flag bundles against per-flag deny entries. Previously, a deny rule for `-sU` would not block `-sUT` because the comparison required an exact token match. The fix applies bundle expansion only for short-form tokens (single dash, all-alpha, at most 4 characters) so long-form single-dash flags used by Go tools (e.g. `-list`, `-output`) still require an exact match.
- **Permalink page app title displayed inside stale green border** — the generic `header h1` rule in `base.css` applied a green border, gradient background, and padding to the permalink page's `<h1 class="permalink-title">` element, a style that had been removed from the main shell UI. The permalink-specific overrides in `components.css` covered font and color but not border, background, or padding. Fixed by adding explicit resets to `body.permalink-page .permalink-title`.
- **HTML export command text rendered in uppercase** — `buildTerminalExportStyles` in `export_html.js` applied `text-transform: uppercase` to `.export-meta`, making the command line appear all-caps in every downloaded HTML file from both the main UI and the permalink page. Removed the CSS property; command text now preserves its original casing, which is important for case-sensitive flags.
- **PDF export command and title text converted to uppercase** — both `tabs.js` and `permalink.html` called `.toUpperCase()` on the app name and meta line before passing them to jsPDF. Unlike CSS `text-transform`, this mutates the string, so commands with case-sensitive flags were corrupted. Removed the `.toUpperCase()` calls from all call sites.
- **PDF export backgrounds swapped vs HTML export** — the PDF header was filled with `colors.surface` (lighter) while the content body used `colors.bg` (darker), which is the reverse of the HTML export layout. Fixed by swapping the fills so the header uses `colors.bg` and the content body uses `colors.surface`, matching the HTML output.
- **HTML export had visible blank lines between output lines** — `buildExportLinesHtml` in `export_html.js` joined `<span class="line">` elements with `'\n'`. Because the container has `white-space: pre-wrap`, those literal newlines between block-level spans rendered as visible blank lines in the downloaded HTML file. Fixed by joining with `''` instead.
- **PDF and HTML export included timestamp/line-number prefix column regardless of terminal toggle state** — `exportTabHtml` used `line.tsC || ''` unconditionally, and `exportTabPdf` read `line.tsC` unconditionally, so exports showed timestamps even when the terminal had both timestamps and line numbers disabled. A new `_exportPrefix(line, index)` helper reads the same `tsMode`/`lnMode` globals that govern the live terminal display, so exports now match what is visible in the tab.
- **PDF export app name font too large and over-spaced** — the heading was rendered at 14pt with `setCharSpace(4)`, producing a heavier and more spread appearance than the HTML export's JetBrains Mono 300 at 20px with CSS `letter-spacing: 4px`. Fixed: 13pt with `setCharSpace(2)`.
- **PDF export header line spacing too cramped** — the baseline-to-baseline gaps between the heading, meta line, and badge row were 6pt and 5pt. Fixed: 20pt title-to-meta gap and 14pt meta-to-badge gap.
- **PDF export badge border unevenly padded** — the badge rect was positioned using the full em square above the text baseline, leaving too much space above the text and not enough below. Fixed using cap height (`hBadgePt × 0.72`) and descender height (`hBadgePt × 0.20`) for visually even vertical padding.
- **Permalink export files showed raw ISO 8601 timestamp** — the `created` field from the server is an ISO 8601 string (e.g. `2026-04-16T07:57:31.474893+00:00`). Both `saveHtml()` and `savePdf()` in `permalink.html` passed it directly into the export header. Now formatted with `new Date(created).toLocaleString()` so the timestamp reads like `4/16/2026, 7:57:31 AM` in the exported file.
- **Permalink history page showed only `$` instead of the full prompt echo** — when `appendPromptNewline` stored a trailing empty prompt entry (`{text: "$", cls: "prompt-echo"}`) in the command history, the `has_prompt_echo` check in `_normalize_permalink_lines` was satisfied by that bare entry, preventing the synthetic `$ {label}` command header from being added to the permalink output. Fixed by requiring the prompt-echo entry to have non-whitespace content after the first word, so a bare `$` no longer suppresses the header.
- **HTML exports used a hardcoded 13px font instead of the terminal font size** — `buildTerminalExportStyles` in `export_html.js` applied `font-size: 13px` and `line-height: 1.65` to the export body, ignoring the `--terminal-font-size` and `--terminal-line-height` CSS variables that are captured into the exported document's `:root` block. The export body now uses `font-size: var(--terminal-font-size, 14px)` and `line-height: var(--terminal-line-height, 1.65)` so the exported font matches the live terminal.
- **`session-token set` now validates the token exists on this server** — any `tok_` string was accepted by the client without checking whether the server had actually issued it, so a typo or token from a different instance would silently activate and all subsequent API calls would return empty results. A new `POST /session/token/verify` endpoint returns `{"ok": true, "exists": true/false}`; the terminal `session-token set` command and the options-modal Set flow now call it before applying the token. UUIDs (anonymous sessions) always pass. `/session/migrate` applies the same check on its `to_session_id` field, rejecting unissued `tok_` destinations with 400 before any data is moved. The verification is fail-closed: a network error or non-OK server response blocks the switch rather than allowing an unverified token through, so transient server issues cannot silently strand the user on a bogus identity.
- **Cross-tab session switch now refreshes history, chips, and the options panel** — the `storage` event listener in `session.js` only updated `SESSION_ID` in passive tabs; it did not reload the chip bar, history panel, or options-panel token display. After a session token is set or cleared in any tab, the listener now calls `reloadSessionHistory()` (if defined) and `_updateOptionsSessionTokenStatus()` (if defined) so the passive tab's UI reflects the new session identity immediately without a page reload.
- **Options panel session token buttons incorrect on mobile** — the Rotate and Clear buttons were visible on mobile even when no session token was set, because `_updateOptionsSessionTokenStatus()` was only called from a `click` listener on the desktop options button. On mobile, the panel opens via `openOptions()` directly, which never called it. Moved the call into `openOptions()` so it runs regardless of how the panel is opened; removed the now-redundant standalone listener.
- **Diagnostics page mobile breakpoint did not match the shell's mobile-mode threshold** — the back button and mobile header chrome on `/diag` were controlled by `@media (max-width: 760px)`, while the shell's `useMobileTerminalViewportMode()` switches into mobile mode at `max-width: 900px` combined with touch/coarse-pointer detection. This meant a touch device between 761px and 900px (where the shell is in mobile mode) showed no back button on the diagnostics page, while a narrow desktop window under 760px showed the mobile back button even though the shell stayed in desktop mode. The responsive collapse block in `diag.css` has been split: grid and column layout continue to collapse at `max-width: 760px` for all device types, while the mobile chrome (back button, header layout overrides) now activates at `@media (max-width: 900px) and (pointer: coarse)` to match the same width + touch criteria the shell uses.

Without this, scanner tools like `wpscan` that exit with code 3 as a normal status code (meaning "potentially interesting findings") could crash the server: when the entire subprocess chain receives SIGTERM simultaneously, intermediate parents may die before the leaf tool, orphaning it to PID 1 (Gunicorn). Gunicorn's `reap_workers()` collects the orphan via `waitpid(-1)` and interprets `exit(3)` as `WORKER_BOOT_ERROR`, halting the server. With `init: true`, tini is PID 1, Gunicorn is PID 2+, and orphaned scanner processes are silently reaped by tini.

The new implementation also handles combined short flags (e.g. `-sU` is caught by a `!nmap -sU` deny entry) correctly.
- **Draft input corrupted when switching tabs with a keyboard shortcut** — pressing a keyboard shortcut that contains the Tab key (such as `Alt+Tab`) while `cmdInput` had focus triggered the autocomplete accept handler (`acAccept`) because the `Tab` key case in the `cmdInput` keydown handler had no modifier guard. On the next switch, that tab's draft was restored as the stale suggestion instead of the original command. Fixed by adding `!e.altKey && !e.ctrlKey && !e.metaKey` guards to the `Tab` handler in `controller.js` so modifier-key combinations no longer trigger autocomplete logic.

ArrowDown/Up and Ctrl+R navigate the list and fill the input with the highlighted match; Tab accepts without running; Enter accepts and runs.

Escape and Ctrl+G still restore the original draft.
- **Mobile edit-bar hold-to-repeat** — the left and right cursor-move buttons in the mobile composer now repeat while held: a 400 ms initial delay starts the repeat, which then fires every 60 ms. Document-level `pointerup`, `pointercancel`, `touchend`, and `touchcancel` safety listeners ensure the timer is always cancelled even if the pointer leaves the button while held. Previously a single tap moved the cursor one position with no repeat.

The `pointerdown` and `touchstart` handlers now return early (without calling `preventDefault` or `focus`) when the input is already focused, so the browser can place the cursor at the tapped position normally. A `click` handler additionally captures the tapped cursor position and restores it on the next tick if a downstream synchronisation call moves it to end.

The row handlers now exclude the mobile input element as a valid trigger target.

A catch-all at the end of the `document` keydown handler focuses `cmdInput`, inserts the character via `replaceCmdRange`, and calls `e.preventDefault()`. The catch-all is skipped when a modal is open, when the target is a button or link, or when the target is an editable field.

- **Composer state Phase 3** — focus and selection changes now publish directly into `composerState`, and the last hidden-input mirroring helper was removed. Mobile focus, desktop focus, and selectionchange events now update shared composer state instead of pushing DOM state from one input into the other.

This provides an OS-level backstop for tools that could reach internal endpoints even if they bypass command validation (e.g. scripting languages, `nc`). `iptables` is now installed in the Dockerfile. The rule uses `|| true` so startup is not aborted in environments where `xt_owner` is unavailable.

A `.env` file is included in the repo with the default (`APP_PORT=8888`); edit it to move the whole stack to a different port. The inline `:-8888` fallbacks in `docker-compose.yml` and `ARG APP_PORT=8888` in `Dockerfile` are safety nets only for bare `docker build` runs that bypass Compose. When running `docker compose -f examples/docker-compose.standalone.yml` from the project root, pass `--env-file .env` explicitly because Compose looks for `.env` in the same directory as the compose file.
- **Playwright e2e test commands** — all Playwright specs that previously used `curl http://localhost:5001/...` as the test command were updated to use built-in fake shell commands (`hostname`, `date`, `uptime`, `whoami`, etc.). The `curl localhost` pattern triggers the `_LOOPBACK_RE` block added in this release, causing those tests to fail immediately. Fake commands bypass both the allowlist check and the loopback block because `/run` resolves them before calling `is_command_allowed()`. Tests that run two consecutive commands back-to-back (`clear all history` in `history.spec.js`, `Delete Non-Favorites` in `history.spec.js`, `clearing history` in `failure-paths.spec.js`) also received a `page.waitForTimeout(1200)` between commands: fake commands complete in ~5ms, so without the gap both runs land in the same 1-second rate-limit window under the default `rate_limit_per_second: 5` config. The `history.spec.js` `beforeEach` also gained a `page.reload()` after `localStorage.clear()` to ensure a clean state on each run.

Host browser and curl access arrives as the Docker bridge gateway IP instead, so `172.16.0.0/12` is the correct scope for operator access.

The browser template and jsdom harness both load the controller after `app.js`, preserving the existing classic-script globals while separating the composition root from the reusable helpers.

The browser template and jsdom harness load that helper layer explicitly before the DOM-bound modules that depend on it, which keeps the classic-script architecture intact while reducing implicit ordering requirements.

## [1.0] — initial release

### Added
- **Real-time command execution** — bash commands streamed to the browser over SSE; output rendered with ANSI color support via `ansi_up`
- **Multi-tab interface** — open multiple independent tabs, each with its own status, output, and kill button
- **Command allow/deny rules** — `allowed_commands.txt` whitelist with group labels; deny-prefix rules block dangerous flags (e.g. `--output`, `-oN`); `/dev/null` output flag allowed explicitly
- **Process kill** — per-tab ■ Kill button sends SIGTERM to the entire process group; confirmation modal prevents accidental kills; PID tracked in SQLite to avoid multi-worker race conditions
- **Run history panel** — last 50 runs per session stored in SQLite; load any result into a new tab; delete individual runs or clear all; session-isolated so users only see their own history
- **Permalinks** — share button saves a full tab snapshot to SQLite and returns a shareable URL; single-run permalinks also available from the history panel; styled HTML view with copy / save .txt options
- **Output search** — in-terminal search with case-sensitive and regex toggle, prev/next navigation, match count
- **Config YAML** — operator-configurable settings: `app_name`, `motd`, `default_theme`, `rate_limit_per_minute/second`, `command_timeout_seconds`, `max_output_lines`, `max_tabs`, `history_panel_limit`, `permalink_retention_days`, `heartbeat_interval_seconds`
- **Rate limiting** — Flask-Limiter backed by Redis (multi-worker safe); falls back to in-process memory if Redis is unavailable; real client IP auto-detected from `X-Forwarded-For` when valid, otherwise direct connection IP
- **Redis process tracking** — active PIDs stored in Redis (or in-process dict) so any Gunicorn worker can kill a process started by a different worker
- **Security model** — two non-root users: `appuser` runs the Flask/Gunicorn process; `scanner` runs all user commands; filesystem mounted read-only except `/tmp`; `sudo kill` used for cross-user SIGTERM
- **Gunicorn WSGI** — production server with configurable timeout (3600 s); heartbeat SSE comments prevent nginx/browser idle disconnects
- **Custom FAQ** — operator-supplied `faq.yaml` entries append after the built-in FAQ; lightweight markup and clickable command chips load commands directly into the input bar
- **Custom autocomplete** — `auto_complete.txt` drives the command input dropdown
- **MOTD** — optional message-of-the-day rendered in the header area
- **Theme selector** — named theme variants selected from the options modal; preference persisted in localStorage; operator can set `default_theme` to a registry filename if they want a preferred startup variant
- **Mobile menu** — hamburger menu exposes search, history, timestamps, theme, and FAQ on small screens
- **Docker support** — multi-stage Dockerfile; `docker-compose.yml` with Redis sidecar; health checks via `/health` endpoint; read-only root filesystem with tmpfs mounts
- **GitLab CI pipeline** — lint (flake8) and test (pytest) stages run on every push
- **Security tools** — nmap, masscan, naabu, httpx, nuclei, subfinder, dnsx, katana, nikto, wapiti3, wpscan, mtr (report mode), dig, host, whois, curl, ffuf, gobuster, feroxbuster and more
- **SecLists wordlists** — full collection installed at `/usr/share/wordlists/seclists/`
- **ARCHITECTURE.md** — documents the system design, data flow, module structure, and database schema
- **Unit test suite** — pytest tests covering command validation, route behavior, rate limiting, and deny-rule logic; lint enforced via flake8
- **ARIA accessibility** — labelled inputs and buttons throughout the UI

# Changelog

All notable changes to darklab shell are documented here.

---

## [1.5] ŌĆö Unreleased

### Fixed

- **`session-token revoke` now enforces invalidation at the API layer** ŌĆö previously, revoking a token deleted it from `session_tokens` but the main data routes still trusted any `X-Session-ID` header value unconditionally via `get_session_id()`, so a browser or script that kept sending the old token could still read and write the same runs, snapshots, and stars. `get_session_id()` now validates `tok_` tokens against `session_tokens` on every request; a revoked or never-issued `tok_` header value is treated as anonymous (returns `""`) so the caller loses access to the session's data. `/session/token/info` also no longer echoes back a `tok_` header value when no matching row exists ŌĆö both fields return `null`, matching the anonymous case.
- **Desktop notifications no longer expose full command arguments** ŌĆö `_maybeNotify()` previously built the notification title from `'$ ' + command` (truncated to 80 characters), which would include bearer tokens, API keys, auth headers, internal hostnames, or the token value in `session-token revoke <token>`. The title now uses only the command root (first word), e.g., `$ curl` instead of `$ curl -H "Authorization: Bearer sk-..."`. The exit code and elapsed time in the notification body are unaffected.
- **`_load_autocomplete_config` no longer silently drops empty `arg_hints` entries** ŌĆö an `if hints:` guard in the normalization loop treated an empty Python list (`[]`) as falsy and never wrote it to the normalized dict. This meant that entries like `"clear": []` and `"generate": []` were dropped before the config reached the browser, so `directHints` was always `null` for those tokens and the early-return gate never fired. Fixed by writing all `arg_hints` entries unconditionally.
- **`directHints` gate in `autocomplete.js` no longer requires `expects_value` membership** ŌĆö the gate previously fired only when `arg_hints[previousToken]` existed AND the token appeared in `expects_value`. Terminal subcommands like `session-token clear` needed to be in `expects_value` even though they do not consume a value argument. The condition is now `if (directHints !== null)` ŌĆö any `arg_hints` entry (including an empty `[]`) triggers the early return without requiring membership in `expects_value`. Existing commands that use `arg_hints` together with `expects_value` are unaffected; `expects_value` now correctly describes only flags that genuinely consume a value.
- **`session-token generate` and `session-token set` now defer identity switch until after migration** ŌĆö both commands previously switched `SESSION_ID` to the new token before calling `/session/migrate`, so a failed migration left the user on the new token identity while their runs and snapshots remained attached to the old session. The identity switch (and `localStorage` write) now happens only after migration succeeds. If the user declines migration, the switch still happens immediately (the new token activates without migrating history). If migration fails, the old session stays active and an error is shown; the newly generated token is discarded. The options-modal Generate and Set paths follow the same pattern. `rotate` was already atomic; this brings `generate` and `set` in line with it.
- **`_seedLocalStorageStarsToServer` no longer silently drops stars on network failure** ŌĆö failed `POST /session/starred` requests were swallowed and `localStorage.starred` was unconditionally deleted afterward, creating a data-loss window on flaky networks during first token activation. The function now uses `Promise.allSettled`, checks each response's `ok` flag, and writes only the failed commands back to `localStorage`. The key is removed only when every POST succeeds.
- **`_waitForMigrateChoice` modal no longer accumulates stale button listeners** ŌĆö each call to `_waitForMigrateChoice` added new `click` handlers to the yes/no buttons without removing previous ones. If the modal was opened multiple times without a click (e.g. rapid re-triggers), stale `Promise` resolvers could fire on a later open. Fixed by replacing each button with a `cloneNode` copy before adding listeners, so no handler from a prior open survives.
- **`migrated_stars` response count now reflects newly inserted rows, not deleted rows** ŌĆö `/session/migrate` used the `DELETE FROM starred_commands` rowcount for `migrated_stars`. When the destination session already had some of the same starred commands, `INSERT OR IGNORE` skipped them but `DELETE` still removed all source rows, causing the returned count to overstate how many commands were actually migrated. The count now uses the `INSERT OR IGNORE` rowcount, which only counts rows actually written to the destination.
- **Toast notifications now appear above the options modal** ŌĆö the toast `z-index` was 300, below the options overlay at 320, so toasts triggered by session token operations (e.g. "token copied to clipboard") were hidden behind the open modal. Bumped to 400 so toasts render on top of all overlay surfaces.
- **Recent command chips and starred commands now reload immediately after any session token operation** ŌĆö `cmdHistory` (the chip bar data) was populated only at page load and not refreshed when the active session changed. A new `reloadSessionHistory()` function in `history.js` sequences star-cache reload ŌåÆ history fetch ŌåÆ chip-bar hydration ŌåÆ history panel refresh (if open), and is called at every session switch path in `runner.js` and `controller.js`. Previously the chip bar and starred set only reflected the new session after a full page reload.
- **`session-token set` autocomplete no longer inserts the `<token>` placeholder** ŌĆö the positional hint builder in `autocomplete.js` was not passing `insertValue` from YAML items to `_buildAutocompleteItem`, so the display label (`set <token>`) was also used as the inserted text. Both call sites now propagate `label` and `insertValue` from the YAML item; `session-token set` has `insertValue: "set "` in `autocomplete.yaml`, so `Tab` inserts `session-token set ` and leaves the cursor ready for pasting without needing to remove a placeholder.
- **Mobile tab bar now scrolls to keep the active tab fully visible** ŌĆö `shouldScrollActiveTabIntoView()` was explicitly suppressing the `ensureActiveTabVisible()` call on mobile, meaning the active tab could be partially off-screen with no way to see it without manually scrolling. The guard was removed; the call is now unconditional across both desktop and mobile. `setTabLabel()` also now calls `ensureActiveTabVisible()` for the active tab so automatic renames triggered by command submission scroll the tab into view just as manual renames do.
- **Mobile autocomplete dropdown truncated and not scrollable with 3+ items** ŌĆö two separate bugs combined to make the third (and any further) autocomplete item inaccessible on mobile. First, `touchstart` on each dropdown item called `e.preventDefault()`, which blocked the browser's native scroll gesture on the dropdown container ŌĆö so touching the list to scroll it instead triggered the topmost item. Second, `maxHeight` in mobile mode was additionally capped by `available = rect.top - 12` (the vertical space above the composer), which could further clip the dropdown below the height needed to show all items. Fixed by replacing the `touchstart` + immediate-accept pattern with tap detection: `touchstart` now records the finger position passively (no `preventDefault`), and `touchend` only accepts the item if the finger moved less than 10px, letting swipe gestures fall through to native scroll. The secondary `available`-space cap was also removed so the dropdown expands to fit up to 8 items (360px max) without depending on a runtime geometry check.
- **`save txt` and `save html` no longer apply share redaction** ŌĆö local file exports (the "save txt" and "save html" tab actions) were incorrectly running the same redaction pass as permalink sharing, replacing IPs, hostnames, emails, and bearer tokens in downloaded files. Redaction is now scoped exclusively to the permalink share flow where it belongs. `saveTab` now uses the raw exportable lines and `exportTabHtml` uses `rawLines` directly; only `permalinkTab` still calls `_getRedactedLines` before posting to `/share`.
- **Container smoke test now reports all failures instead of stopping after the first** ŌĆö `scripts/container_smoke_test.sh` was passing `-x` to pytest, causing the run to abort on the first failure and leave remaining commands untested. The flag has been removed so the full suite runs and all failures are surfaced together.
- **Reverse-history search now searches full session history** ŌĆö `Ctrl+R` previously searched only the small in-memory recent-commands chip list (capped at `recent_commands_limit`, default 8). It now fetches from the same `/history` endpoint the history drawer uses, applying the search query server-side before the `history_panel_limit` cap so commands from earlier in the session (or from previous days) are always reachable. Results are debounced per keystroke and capped at 10 in the dropdown. The dropdown no longer pre-populates on open ŌĆö it only appears once the first character is typed.
- **Autocomplete examples not shown after accepting a command root** ŌĆö accepting a suggestion via `Tab` or `Enter` (e.g. choosing `dnsenum` while typing `dns`) put the correct value in the prompt but did not show the command's example invocations. The root cause was that `acAccept` called `setComposerValue` before `acHide`: the input event fired synchronously, rendered the examples, and then `acHide` immediately closed them. Swapping the call order (hide first, then set value) lets the input event open the fresh autocomplete state without anything closing it afterward.
- **Example command domains corrected** ŌĆö DNS-oriented tool examples in `autocomplete.yaml` and `scripts/smoke_test_commands.txt` were using `ip.darklab.sh` where they should use `darklab.sh` (a plain domain). Affected tools: `dig`, `nslookup`, `host`, `dnsrecon`, `subfinder`, `dnsx`, `dnsenum`, and `fierce`. Conversely, `nmap` and `curl` examples in `autocomplete.yaml` were using `darklab.sh` where they should use `ip.darklab.sh`. Both corrections are now applied consistently across both files.
- **Diagnostics redaction config rendering** ŌĆö the `/diag` config table now renders numeric zero values as `0` instead of incorrectly treating them as boolean `false`, and the redaction-rule count field is now named `custom_redaction_rule_count` so it is clear the count only refers to operator-defined regex rules, not the built-in baseline.
- **`vitest.config.js` root path resolution** ŌĆö after moving the config to `config/vitest.config.js`, the `root: '..'` option was being resolved relative to the working directory rather than the config file, causing vitest to look one level above the repo root. Fixed by computing the root path from `import.meta.url` so it is always the absolute repo root regardless of invocation directory.
- **naabu forced connect-scan mode** ŌĆö `naabu` commands that do not already specify `-scan-type` or `-st` now have `-scan-type c` injected automatically before reaching the container. Raw SYN scanning requires libpcap and elevated capabilities that are not reliably available inside the container; TCP connect mode (`-sT` equivalent) produces the same open-port results without them.
- **Autocomplete stubs not discoverable via prefix matching** ŌĆö empty specs in the autocomplete YAML (e.g. `whois: {}`) were silently dropped by `_normalize_autocomplete_context` because the guard condition only added a root to the registry when it had flags, arg_hints, or pipe metadata. Typing a partial name such as `whoi` returned an empty dropdown instead of suggesting `whois`. The guard was removed so all declared roots appear in `acContextRegistry` regardless of whether they carry additional spec data.
- **Deny-prefix combined short-flag matching** ŌĆö `_is_denied()` now correctly matches combined POSIX short-flag bundles against per-flag deny entries. Previously, a deny rule for `-sU` would not block `-sUT` because the comparison required an exact token match. The fix applies bundle expansion only for short-form tokens (single dash, all-alpha, at most 4 characters) so long-form single-dash flags used by Go tools (e.g. `-list`, `-output`) still require an exact match.
- **Permalink page app title displayed inside stale green border** ŌĆö the generic `header h1` rule in `base.css` applied a green border, gradient background, and padding to the permalink page's `<h1 class="permalink-title">` element, a style that had been removed from the main shell UI. The permalink-specific overrides in `components.css` covered font and color but not border, background, or padding. Fixed by adding explicit resets to `body.permalink-page .permalink-title`.
- **HTML export command text rendered in uppercase** ŌĆö `buildTerminalExportStyles` in `export_html.js` applied `text-transform: uppercase` to `.export-meta`, making the command line appear all-caps in every downloaded HTML file from both the main UI and the permalink page. Removed the CSS property; command text now preserves its original casing, which is important for case-sensitive flags.
- **PDF export command and title text converted to uppercase** ŌĆö both `tabs.js` and `permalink.html` called `.toUpperCase()` on the app name and meta line before passing them to jsPDF. Unlike CSS `text-transform`, this mutates the string, so commands with case-sensitive flags were corrupted. Removed the `.toUpperCase()` calls from all call sites.
- **PDF export backgrounds swapped vs HTML export** ŌĆö the PDF header was filled with `colors.surface` (lighter) while the content body used `colors.bg` (darker), which is the reverse of the HTML export layout. Fixed by swapping the fills so the header uses `colors.bg` and the content body uses `colors.surface`, matching the HTML output.
- **HTML export had visible blank lines between output lines** ŌĆö `buildExportLinesHtml` in `export_html.js` joined `<span class="line">` elements with `'\n'`. Because the container has `white-space: pre-wrap`, those literal newlines between block-level spans rendered as visible blank lines in the downloaded HTML file. Fixed by joining with `''` instead.
- **PDF and HTML export included timestamp/line-number prefix column regardless of terminal toggle state** ŌĆö `exportTabHtml` used `line.tsC || ''` unconditionally, and `exportTabPdf` read `line.tsC` unconditionally, so exports showed timestamps even when the terminal had both timestamps and line numbers disabled. A new `_exportPrefix(line, index)` helper reads the same `tsMode`/`lnMode` globals that govern the live terminal display, so exports now match what is visible in the tab.
- **PDF export app name font too large and over-spaced** ŌĆö the heading was rendered at 14pt with `setCharSpace(4)`, producing a heavier and more spread appearance than the HTML export's JetBrains Mono 300 at 20px with CSS `letter-spacing: 4px`. Fixed: 13pt with `setCharSpace(2)`.
- **PDF export header line spacing too cramped** ŌĆö the baseline-to-baseline gaps between the heading, meta line, and badge row were 6pt and 5pt. Fixed: 20pt title-to-meta gap and 14pt meta-to-badge gap.
- **PDF export badge border unevenly padded** ŌĆö the badge rect was positioned using the full em square above the text baseline, leaving too much space above the text and not enough below. Fixed using cap height (`hBadgePt ├Ś 0.72`) and descender height (`hBadgePt ├Ś 0.20`) for visually even vertical padding.
- **Permalink export files showed raw ISO 8601 timestamp** ŌĆö the `created` field from the server is an ISO 8601 string (e.g. `2026-04-16T07:57:31.474893+00:00`). Both `saveHtml()` and `savePdf()` in `permalink.html` passed it directly into the export header. Now formatted with `new Date(created).toLocaleString()` so the timestamp reads like `4/16/2026, 7:57:31 AM` in the exported file.
- **Permalink history page showed only `$` instead of the full prompt echo** ŌĆö when `appendPromptNewline` stored a trailing empty prompt entry (`{text: "$", cls: "prompt-echo"}`) in the command history, the `has_prompt_echo` check in `_normalize_permalink_lines` was satisfied by that bare entry, preventing the synthetic `$ {label}` command header from being added to the permalink output. Fixed by requiring the prompt-echo entry to have non-whitespace content after the first word, so a bare `$` no longer suppresses the header.
- **HTML exports used a hardcoded 13px font instead of the terminal font size** ŌĆö `buildTerminalExportStyles` in `export_html.js` applied `font-size: 13px` and `line-height: 1.65` to the export body, ignoring the `--terminal-font-size` and `--terminal-line-height` CSS variables that are captured into the exported document's `:root` block. The export body now uses `font-size: var(--terminal-font-size, 14px)` and `line-height: var(--terminal-line-height, 1.65)` so the exported font matches the live terminal.
- **`session-token set` now validates the token exists on this server** ŌĆö any `tok_` string was accepted by the client without checking whether the server had actually issued it, so a typo or token from a different instance would silently activate and all subsequent API calls would return empty results. A new `POST /session/token/verify` endpoint returns `{"ok": true, "exists": true/false}`; the terminal `session-token set` command and the options-modal Set flow now call it before applying the token. UUIDs (anonymous sessions) always pass. `/session/migrate` applies the same check on its `to_session_id` field, rejecting unissued `tok_` destinations with 400 before any data is moved. The verification is fail-closed: a network error or non-OK server response blocks the switch rather than allowing an unverified token through, so transient server issues cannot silently strand the user on a bogus identity.
- **Cross-tab session switch now refreshes history, chips, and the options panel** ŌĆö the `storage` event listener in `session.js` only updated `SESSION_ID` in passive tabs; it did not reload the chip bar, history panel, or options-panel token display. After a session token is set or cleared in any tab, the listener now calls `reloadSessionHistory()` (if defined) and `_updateOptionsSessionTokenStatus()` (if defined) so the passive tab's UI reflects the new session identity immediately without a page reload.
- **Options panel session token buttons incorrect on mobile** ŌĆö the Rotate and Clear buttons were visible on mobile even when no session token was set, because `_updateOptionsSessionTokenStatus()` was only called from a `click` listener on the desktop options button. On mobile, the panel opens via `openOptions()` directly, which never called it. Moved the call into `openOptions()` so it runs regardless of how the panel is opened; removed the now-redundant standalone listener.
- **Diagnostics page mobile breakpoint did not match the shell's mobile-mode threshold** ŌĆö the back button and mobile header chrome on `/diag` were controlled by `@media (max-width: 760px)`, while the shell's `useMobileTerminalViewportMode()` switches into mobile mode at `max-width: 900px` combined with touch/coarse-pointer detection. This meant a touch device between 761px and 900px (where the shell is in mobile mode) showed no back button on the diagnostics page, while a narrow desktop window under 760px showed the mobile back button even though the shell stayed in desktop mode. The responsive collapse block in `diag.css` has been split: grid and column layout continue to collapse at `max-width: 760px` for all device types, while the mobile chrome (back button, header layout overrides) now activates at `@media (max-width: 900px) and (pointer: coarse)` to match the same width + touch criteria the shell uses.

### Changed

- **Desktop shell chrome rebuilt** ŌĆö the desktop layout is now a single composed surface. The top header, recent-command chip row, and per-tab footer action row have been replaced by a persistent left rail (app title, recent commands, workflows, history, options, theme, FAQ, diag, version footer) and a persistent bottom HUD bar that owns the status pills, search, quick toggles, kill button, and the `share snapshot / copy / save Ō¢Š / clear` actions. The tabbar row now carries tab chrome plus the inline `search / line numbers / timestamps` controls on the right. The mobile layout is handled by its own companion refactor (see the next entry); the `syncMobileShellLayout()` re-parenting path is preserved so the classic tab and composer DOM still feeds the mobile chrome. Implemented as `app/static/css/shell-chrome.css` (651 lines, imported after `welcome.css` in `styles.css`) and `app/static/js/shell_chrome.js` (438 lines, loaded last after `controller.js`) so the rail / HUD / tabbar scaffolding lives outside the per-tab DOM and is not duplicated per tab. HUD action buttons carry `data-action` attributes for stable test selection, and the permalink and diag pages are explicitly scoped out of the desktop header hide so their own `<header class="export-header">` still renders.
- **Mobile shell refactor** ŌĆö the mobile chrome was rebuilt around a dedicated bottom-sheet and pull-up sheet model rather than re-parented desktop affordances. Implemented across six phases on the v1.5 branch and bundled here. `body.mobile-terminal-mode` now uses a normal-flow flex column with a fixed header, scrollable transcript, and an idle peek row that opens a full pull-up **recents sheet** (search, filter chips, starring, copy/permalink/restore per row); the older mobile recent-command chip strip is gone. The hamburger opens a **menu sheet** with two visually grouped sections separated by an inset hairline ŌĆö a session group (`search` ┬Ę `line numbers` ┬Ę `timestamps`) where toggles render their state via `aria-pressed` (on labels turn `--green`), and an overlays group (`history` ┬Ę `workflows` ┬Ę `options` ┬Ę `theme` ┬Ę `FAQ` ┬Ę `diag`) ŌĆö with a centred `darklab shell v1.5` wordmark footer. `timestamps` is an inline expander rather than a tap-cycle: tapping the row toggles a sub-menu listing `off` / `elapsed` / `clock` with radio dots and live hint values (`+1.2s`, `14:03:27`); the sheet stays open while expanding and the sub-menu collapses on every fresh sheet open. The legacy `#mobile-menu` proxy DOM (a hidden mirror of the desktop menu used as a re-parenting source) was removed and `_setMobileActionLabel()` rewired to write directly to the sheet rows. A new mobile **edit-helper row** rides above the composer when the keyboard is open with cursor motion, line jumps, and word-aware deletes; the iOS form-assistant bar is left alone instead of fought. Status and runtime now ride the `RUNNING` pill inline, share/copy/save/clear collapsed into the menu, and the desktop HUD strip is hidden on mobile entirely. All sheet surfaces source colour from existing semantic tokens (`--surface`, `--border`, `--border-bright`, `--text`, `--muted`, `--green`, `--amber`, `--red`) plus a small set of `--theme-*-bg` variables with safe `var(..., fallback)` defaults, so all 17 themes render the new chrome without per-theme rules. A reusable `tests/js/e2e/theme-audit.spec.js` Playwright tool iterates every theme, force-opens each sheet, and reads computed styles to verify WCAG contrast ratios on ten representative pairs (no theme falls below 1.20 on any pair). New CSS lives in `app/static/css/mobile-chrome.css` and new behaviour in `app/static/js/mobile_chrome.js`; sheet visibility uses the `u-hidden` utility with explicit per-class re-overrides where the scoped `.menu-sheet`/`.menu-submenu` rules outrank the base `display: none`.
- **Diagnostics page CSS extracted to `diag.css`** ŌĆö the 312-line inline `<style>` block in `diag.html` has been moved to `app/static/css/diag.css` and loaded as a linked stylesheet. `terminal_export.css` is now also linked from `diag.html` so the `export-header` and `export-header-copy` classes it defines are active on the diagnostics page, establishing the same shared header chrome foundation used by permalink pages. No visual change; this is a structural refactor.
- **`_setMobileActionLabel` centralises mobile menu label updates** ŌĆö `_setTsMode` in `app.js` and `_setLnMode` in `output.js` previously each queried the mobile menu DOM independently to update action button labels. Both now go through a shared `_setMobileActionLabel(action, label)` helper in `app.js`, which holds the `mobileMenu` reference. `output.js` calls it via a `typeof` guard matching the pattern used elsewhere across the codebase.
- **`session-token` subcommands now have per-subcommand autocomplete context** ŌĆö after `session-token set `, the dropdown shows a `<token>` value hint. After `session-token generate `, `session-token clear `, or `session-token rotate `, the dropdown collapses (no further suggestions). Previously, all three terminal subcommands continued to suggest the other available subcommands as if additional arguments were expected. `generate`, `clear`, and `rotate` are removed from `expects_value` since they do not consume a value; only `set` remains.
- **Help flags marked as terminal in `arg_hints` across 24 commands** ŌĆö typing a help flag and then pressing space no longer pops the autocomplete dropdown. The following flags are now marked with `[]` in `arg_hints`: `-h` for nmap, ffuf, curl, dig, nuclei, ping, mtr, wget, dnsrecon, dnsenum, dnsx, fierce, subfinder, pd-httpx, wpscan, gobuster, fping, sslscan; `--help` for traceroute, testssl, sslyze, wafw00f, masscan, rustscan; `-help` for naabu and katana. Excluded intentionally: `nikto -h` (target host, not help), `df -h` and `free -h` (human-readable output modifier, combinable with other flags).
- **Repository structure cleanup** ŌĆö tool and test config files that were cluttering the repo root have been consolidated into a `config/` subdirectory (`eslint.config.js`, `hadolint.yaml`, `yamllint.yml`, `vitest.config.js`, `playwright.config.js`, `playwright.parallel.config.js`, `playwright.shared.js`, `playwright.demo.config.js`, `playwright.demo.mobile.config.js`). The `docs/` directory has been renamed to `assets/` to better reflect its purpose as a media asset store rather than a documentation folder. The `scripts/node/` subdirectory has been removed and its single file (`capture_output_for_smoke_test.mjs`) moved up to `scripts/` root to avoid a language-named subfolder for one file. All internal references and `package.json` scripts have been updated.
- **Container smoke test is now driven by `autocomplete.yaml` examples** ŌĆö the command corpus for both `scripts/container_smoke_test.sh` and `scripts/capture_container_smoke_test_outputs.sh` is now sourced from `context.<root>.examples[].value` in `app/conf/autocomplete.yaml` instead of the manually maintained `scripts/smoke_test_commands.txt`. This ties the regression suite directly to the commands the shell suggests to users: adding or removing an example automatically adds or removes it from the smoke test. Any entry in `tests/py/fixtures/container_smoke_test-expectations.json` whose command does not appear in `autocomplete.yaml` examples is considered stale.
- **Capture script no longer kills long-running commands early** ŌĆö the capture script previously sent SIGTERM to `nmap`, `masscan`, `nikto`, `testssl`, and `nuclei` as soon as early-output markers appeared, resulting in synthetic `-15` exit codes that masked real command failures and truncated output before useful scan results. The early-stop mechanism has been removed. Commands now run to completion (or until `--command-timeout-ms`), producing real exit codes and full output that operators can use to choose meaningful `expected_text` anchors such as actual port scan results instead of just banner lines.
- **Help examples added as first example for all tool commands** ŌĆö every command entry in `app/conf/autocomplete.yaml` that declares a help flag (`-h`, `--help`, or `-help`) now has that help invocation as its first example. This surfaces the help output as a suggested starting point for users unfamiliar with a tool and ensures the tool's help flag is smoke-tested on every run.
- **Docker container networking tuning** ŌĆö `examples/docker-compose.prod.yml` now raises the per-process file-descriptor ceiling to 65 535 via `ulimits.nofile` and adds network-namespace-local sysctls (`net.ipv4.ip_local_port_range`, `net.ipv4.tcp_tw_reuse`, `net.ipv4.tcp_fin_timeout`, `net.ipv4.tcp_max_tw_buckets`) to the shell container. This gives tools like `naabu` and `masscan` enough source ports and socket headroom to run wider port-range scans without silently dropping connections.
- **`man <built-in>` autocomplete coverage** ŌĆö the `man` command's positional hints in `autocomplete.yaml` now include every documented built-in command root so typing `man ` shows both real tool man pages and all built-in commands in the dropdown. The duplicate `nmap` entry was also removed.
- **Documentation set cleanup** ŌĆö `README.md`, `ARCHITECTURE.md`, and `tests/README.md` now have clearer ownership with less overlap, `ARCHITECTURE.md` has been split from the new `DECISIONS.md`, the README now has a top-level documentation map, and the testing doc now reads as a proper handbook plus full appendix rather than a loose inventory.
- **Built-in command presentation** ŌĆö the shellŌĆÖs built-in commands now read more like intentional terminal output instead of plain text dumps. `help`, `shortcuts`, `status`, `limits`, `retention`, `faq`, `history`, `last`, `version`, `whoami`, `env`, `ls`, `ps`, `ip a`, `route`, `jobs`, `df -h`, `free -h`, and plain `uname` now use clearer section headers, shell-style summaries, or session-aware output where appropriate. The snark pools for `sudo`, `reboot`, `poweroff` / `halt` / `shutdown now`, `su`, and the guarded `rm -fr /` / `rm -rf /` variants were also refreshed to use drier, more technical sarcasm while keeping the responses random.

## [1.5] — Unreleased

### Added

- **`Alt+/` FAQ shortcut plus tooltip audit across every chrome button** — the FAQ overlay was the last chrome surface without a keyboard path, and several chrome buttons had no tooltip coverage.
  - **Why:** FAQ was the only overlay you could not open from the keyboard; separately, an audit of desktop chrome tooltips found several buttons (`Kill`, `share snapshot`, `copy`, `save`, `clear`, `new-tab-btn`, per-tab `✕`) with no `title` at all, plus a stale `(Alt+F)` on `search-toggle-btn`.
  - **What:** new `Alt+/` chord bound via `handleChromeShortcut()` in `app.js` (matches `e.code === 'Slash'`, `e.key === '/'`, and the macOS Option glyph `÷`); added as the seventh entry in the `UI` section of `_CURRENT_SHORTCUTS` (`app/fake_commands.py`) so it surfaces in both `shortcuts` and the `?` overlay automatically. Tooltip audit: rail nav items (`history`, `options`, `theme`, `FAQ`) list the matching Alt chord; HUD action buttons gained `title` attrs with shortcut hints (`Option+P / Alt+P`, `Option+Shift+C`, `Ctrl+L`); `new-tab-btn` advertises `(Option+T / Alt+T)`; per-tab close advertises `(Option+W / Alt+W when active)`; stale `(Alt+F)` on search toggle corrected to `(Alt+S)`. Mobile buttons that previously had only `aria-label` (`hamburger-btn`, `mobile-run-btn`, `mobile-kill-btn`) gained matching `title` attrs for tablet/stylus tooltips. Rail section headers (Recent, Workflows) gained titles for their collapse/expand behaviour.
  - **Tests:** one new Playwright case `Alt+/ toggles the FAQ overlay from the composer` in `tests/js/e2e/shortcuts.spec.js`.
- **Desktop chrome keyboard shortcuts** — six new Alt chords toggle the rail and the four major overlay surfaces without touching the mouse.
  - **Why:** desktop chrome surfaces (rail, search bar, history drawer, options panel, theme selector, workflows panel) all required mouse interaction to open and close; there was no keyboard path for any of them.
  - **What:** six toggles — `Alt+\` (rail), `Alt+S` (search), `Alt+H` (history), `Alt+,` (options), `Alt+Shift+T` (theme), `Alt+G` (workflows). `Alt+S` not `Alt+F` for search because the composer owns `Alt+F` as readline word-forward. Each chord is a true toggle — pressing while open closes the surface; opening while another is open swaps cleanly via the existing `_closeMajorOverlays()` path. Bindings matched through a new `handleChromeShortcut()` dispatcher in `app.js` that accepts both `e.code` (for punctuation keys like `Backslash` and `Comma`) and the macOS Option-letter glyphs (`˙` for Alt+H, `ƒ` for Alt+F, `«` for Alt+\, etc.), so any keyboard layout lands on the same result. Called from the document keydown chain in `controller.js` after `handleTabShortcut` and `handleActionShortcut`, and allowed through the major-overlay early-return gate so the same chord that opened a surface can close it again. `handleTabShortcut` was tightened with `!e.shiftKey` guards on its Alt+T and Alt+W branches so Alt+Shift+T falls through to the chrome handler instead of being eaten by the tab dispatcher. `shell_chrome.js` exports `toggleRailCollapsed` from its IIFE so Alt+\ can drive it. Toggle-close paths call `hideHistoryPanel()` / `closeOptions()` / `closeThemeSelector()` / `closeWorkflows()` directly rather than re-using the header-button `.click()` paths — those handlers call `_closeMajorOverlays()` first and would close-then-reopen the same surface. Button tooltips in `templates/index.html` advertise each binding; bindings added to `_CURRENT_SHORTCUTS` so they surface in `shortcuts` and the `?` overlay.
  - **Tests:** seven new cases in `tests/js/e2e/shortcuts.spec.js` under `test.describe('desktop chrome keyboard shortcuts')` covering each chord's toggle behavior plus a negative test confirming chrome shortcuts are ignored while the composer is focused.
- **Dedicated keyboard-shortcuts overlay with a single source of truth** — pressing `?` from anywhere on the page opens a transparent, keyboard-first overlay listing every shortcut.
  - **Why:** keyboard shortcuts were listed only in the FAQ modal (hard to find mid-session) and the `shortcuts` terminal command (requires a runnable prompt). Two reference surfaces meant drift was possible between what the FAQ said and what the terminal command reported.
  - **What:** `?` from anywhere opens a transparent overlay; binding works from the command prompt when empty, but once any text is in the prompt (or any other input) `?` types normally so args like `curl "…?foo=bar"` still work. Both the `shortcuts` built-in command and the overlay read from a new `GET /shortcuts` JSON endpoint backed by `get_current_shortcuts()` in `fake_commands.py`, so reference cannot drift. Overlay lists its own `?` trigger first so the binding is self-documenting. The `?` handler skips modifier chords (Ctrl/Meta/Alt) and the welcome-animation active state — pressing `?` during welcome continues to settle the intro. Escape and backdrop click close the overlay. FAQ's long shortcut list reduced to a short pointer (`press ?` / `run shortcuts`).
  - **Tests:** five new Playwright cases under `test.describe('? keyboard-shortcuts overlay')` in `tests/js/e2e/shortcuts.spec.js` — open when no input focused, Escape closes, opens from empty prompt, types normally when prompt has text, and `overlay and shortcuts built-in share the same source`. Four new pytest cases under `TestShortcutsRoute` in `tests/py/test_routes.py` including `test_matches_shortcuts_builtin_source` pinning the endpoint payload to the `shortcuts` built-in source.
- **UI screenshot capture pipeline for design review, theming, and visual QA** — standalone Playwright projects drive a curated walk through desktop and mobile, emitting labelled PNG packs and per-UI manifest JSON.
  - **Why:** design review and theme QA were ad-hoc — there was no way to produce a consistent, labelled set of screenshots across all 17 themes for both UIs. Manual capture was slow and inconsistent; history/status/recents states were unseeded, so a capture of an empty instance did not show the chrome the way operators would see it in production.
  - **What:** two new Playwright projects (`tests/js/e2e/ui-capture.desktop.capture.js`, `tests/js/e2e/ui-capture.mobile.capture.js`) plus a shared `ui_capture_shared.js` scene registry. Runs via `scripts/capture_ui_screenshots.sh` (`npm run capture:ui-screenshots`) with optional `--ui desktop|mobile`, `--theme <slug>|all`, and output-directory overrides; defaults write to `test-results/ui-capture/`. Captures are gated by `RUN_CAPTURE=1` and live under dedicated configs (`config/playwright.capture.desktop.config.js`, `config/playwright.capture.mobile.config.js`) so they never run in the normal `test:e2e` suite. A new `scripts/seed_history.py` populates `history.db` with ~70 realistic runs under a UUID or `tok_` session so HUD status, `/diag`, recents, and history-heavy states render the way they would in production; the capture harness pre-seeds token `tok_cafebabecafebabecafebabecafebabe` into each capture slot's ephemeral data directory. New `APP_FAKE_REDIS=1` env var in `app/process.py` installs an in-memory Redis subset (`_FakeRedisClient` with `ping`/`set`/`get`/`getdel`/`sadd`/`smembers`/`expire`/`srem`/`delete` plus per-key TTL purging) so capture runs exercise Redis-dependent paths without a real Redis instance. `scripts/playwright/run_e2e_server.sh` detects the `capture-*` slot prefix and opts into both the seeder and the fake client. Seeder refuses to write from the host by default (container/host SQLite can disagree on FTS5 internals — see the `sqlite_cross_version_fts` memory rule).
  - **Tests:** capture runs are themselves the verification; `tests/README.md` documents the seeder as a manual-QA helper for the history drawer, reverse-i-search, filters, and token-migration flows.

Padding reduces the content-box width so `white-space: pre-wrap` wraps at the correct edge. ## [1.0] — initial release

### Added
- **Real-time command execution** — bash commands streamed to the browser over SSE; output rendered with ANSI color support via `ansi_up`
- **Multi-tab interface** — open multiple independent tabs, each with its own status, output, and kill button
- **Command allow/deny rules** — `allowed_commands.txt` whitelist with group labels; deny-prefix rules block dangerous flags (e.g. `--output`, `-oN`); `/dev/null` output flag allowed explicitly
- **Process kill** — per-tab ■ Kill button sends SIGTERM to the entire process group; confirmation modal prevents accidental kills; PID tracked in SQLite to avoid multi-worker race conditions
- **Run history panel** — last 50 runs per session stored in SQLite; load any result into a new tab; delete individual runs or clear all; session-isolated so users only see their own history
- **Permalinks** — share button saves a full tab snapshot to SQLite and returns a shareable URL; single-run permalinks also available from the history panel; styled HTML view with copy / save .txt options
- **Output search** — in-terminal search with case-sensitive and regex toggle, prev/next navigation, match count
- **Config YAML** — operator-configurable settings: `app_name`, `motd`, `default_theme`, `rate_limit_per_minute/second`, `command_timeout_seconds`, `max_output_lines`, `max_tabs`, `history_panel_limit`, `permalink_retention_days`, `heartbeat_interval_seconds`
- **Rate limiting** — Flask-Limiter backed by Redis (multi-worker safe); falls back to in-process memory if Redis is unavailable; real client IP auto-detected from `X-Forwarded-For` when valid, otherwise direct connection IP
- **Redis process tracking** — active PIDs stored in Redis (or in-process dict) so any Gunicorn worker can kill a process started by a different worker
- **Security model** — two non-root users: `appuser` runs the Flask/Gunicorn process; `scanner` runs all user commands; filesystem mounted read-only except `/tmp`; `sudo kill` used for cross-user SIGTERM
- **Gunicorn WSGI** — production server with configurable timeout (3600 s); heartbeat SSE comments prevent nginx/browser idle disconnects
- **Custom FAQ** — operator-supplied `faq.yaml` entries append after the built-in FAQ; lightweight markup and clickable command chips load commands directly into the input bar
- **Custom autocomplete** — `auto_complete.txt` drives the command input dropdown
- **MOTD** — optional message-of-the-day rendered in the header area
- **Theme selector** — named theme variants selected from the options modal; preference persisted in localStorage; operator can set `default_theme` to a registry filename if they want a preferred startup variant
- **Mobile menu** — hamburger menu exposes search, history, timestamps, theme, and FAQ on small screens
- **Docker support** — multi-stage Dockerfile; `docker-compose.yml` with Redis sidecar; health checks via `/health` endpoint; read-only root filesystem with tmpfs mounts
- **GitLab CI pipeline** — lint (flake8) and test (pytest) stages run on every push
- **Security tools** — nmap, masscan, naabu, httpx, nuclei, subfinder, dnsx, katana, nikto, wapiti3, wpscan, mtr (report mode), dig, host, whois, curl, ffuf, gobuster, feroxbuster and more
- **SecLists wordlists** — full collection installed at `/usr/share/wordlists/seclists/`
- **ARCHITECTURE.md** — documents the system design, data flow, module structure, and database schema
- **Unit test suite** — pytest tests covering command validation, route behavior, rate limiting, and deny-rule logic; lint enforced via flake8
- **ARIA accessibility** — labelled inputs and buttons throughout the UI
